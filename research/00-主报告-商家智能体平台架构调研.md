## 1. 执行摘要（先看这 10 条判断）

1. **你的整体方向是对的，而且被市场领导者反向验证了。**"通用 Agent 基座 + 按职位接入不同 skill + 连接企业已有系统"正是 Glean、Copilot Studio、腾讯 ADP、钉钉悟空等所有严肃玩家的共同形态。不必推翻重来。

2. **但"FSE 上门建 skill"要重新定位。** Salesforce 在 2025 年被迫增设 "forward-deployed engineers" 去救火停滞的部署（6 个月→3 周全靠上门），说明**企业 Agent 落地本质是"重服务"**——这既印证你的直觉，也警示：FSE 是成本中心与护城河，但要用**平台化工具 + 服务商生态**放大，而不是靠人力堆。腾讯乐享自己就走 self-service + 服务商，不自养驻场 FSE。

3. **权限隔离是第一硬需求，且有明确的行业标准答案：** 分层授权 = RBAC（粗粒度守门）+ **ReBAC（细粒度核心，用 OpenFGA）** + ABAC（上下文）。这一点由两条独立证据交叉验证：我们的授权专项调研推荐 OpenFGA，而权限模型最严谨的竞品 Moveworks 恰好也用 OpenFGA/Zanzibar 式 ReBAC。

4. **"Agent 权限必须 ≤ 当前对话用户权限"是不可动摇的铁律。** 用 OAuth 2.0 Token Exchange（RFC 8693）做身份传递（`act` claim：用户=subject，agent=actor）。绝不能用一个"胖服务账号"跑所有人的请求——Salesforce 的 ForcedLeak 漏洞（CVSS 9.4）就是这么泄的。

5. **检索必须"先过滤，后喂给大模型"（filter-before-grounding）。** 所有权限感知检索的标杆（Glean/Amazon Q/Google）都是：解析用户身份+组 → 与每篇文档 ACL 求交集 → 只有授权内容才进入 LLM 上下文。绝不能让大模型"看到全部再自我约束"——大模型不负责安全。

6. **知识库分"个人 + 企业"两个 scope，但用同一套基建实现**（元数据 scope，而非两套系统）。向量库首选 **Qdrant（Apache 2.0）**，已重度用 Postgres 则 pgvector。ACL 作为 chunk 元数据在服务端强制预过滤。GraphRAG 暂缓。

7. **桌面客户端首选 Tauri v2**（deny-by-default 的能力/路径沙箱最适合"能碰本地文件"的 agent），团队纯 JS 则 Electron。本地文件操作等"近身"工具在本地跑，业务系统/RAG/敏感推理在后端跑，两边都用 **MCP** 统一表达。

8. **MCP 已是事实标准**（连 Salesforce Agentforce 3、OpenAI、Google 都纳入）。企业连接器一律做成 MCP Server；MCP 本身只是 OAuth 2.1 资源服务器，**RBAC 要我们自己在"scope→角色→可用工具"上建**。

9. **别赌自主 Agent。** Salesforce 自研基准 CRMArena-Pro：多轮任务成功率仅 ~35%。关键业务路径用**确定性工作流/状态机**做骨架，LLM 只在节点内理解与生成，破坏性操作强制人在环（human-in-the-loop）。

10. **市场有泡沫（Gartner：2027 前 >40% agentic 项目将被取消）。** 差异化不在"更炫的 agent"，而在**诚实可核验的指标 + 严格权限隔离 + 落地服务能力**。这三点应是产品立身之本。

## 2. 对你原始设想的逐条评估

你的原始设想分五块，我逐条给出"对/需调整/风险"判断：

### 2.1 "做一个通用 Agent，帮企业接入不同职位的 skill" —— ✅ 方向对，需精化
- **对的部分**：通用基座 + 可插拔 skill 是正确形态，全行业收敛于此。
- **需精化**：不要一个"全能通用 Agent"，而要**一个通用编排层 + 一组窄而可验证的 skill/工具**。CRMArena-Pro 证明开放式自主多轮任务成功率仅约 35%。skill 越窄、越确定、越可测，成功率越高。
- **建议**："通用"体现在**流程基座、权限框架、知识库框架、连接器框架**的复用；"专用"体现在每家企业/每个职位的 skill 配置与数据接入。

### 2.2 "生产流程有差异，但流程基座通用" —— ✅ 成立，这是产品化的关键
- 这是你商业模式能规模化的核心假设，成立。做法：把"订单进度查询""生产计划状态""物料齐套检查"等**抽象成可配置的场景模板（scenario template）**，每个模板 = 一段确定性工作流 + 若干 skill 绑定 + 提示词 + 所需权限声明。
- 新客户上线 = 选模板 + 填连接参数 + 映射权限，而非从零写代码。这正是腾讯 ADP（FAQ agent 1–2 周、多场景 4–6 周）的做法。

### 2.3 "权限必须隔离好，manager 与普通职员不同" —— ✅ 第一优先级，方案见第 6 节
- 这是**最重要且最容易做错**的部分。行业标准答案明确（分层授权 + ReBAC + 身份传递 + filter-before-grounding），第 6 节详述。
- 关键认知升级：**权限不该是"manager 一套 / 员工一套"的静态两档**，而应是**从数据 ACL 自然涌现（emergent from ACLs）**——同一个问题，销售和厂长检索到的数据集天然不同，因为他们对底层记录的授权不同。静态角色档会随组织变化漂移。

### 2.4 "落地是 Windows app，支持读写本地文件" —— ✅ 可行，方案见第 8 节
- Tauri v2 的路径沙箱天然适合"能碰本地文件"的 agent。破坏性本地操作（删除/覆盖/批量移动）强制二次确认 + dry-run + 可撤销。
- 本地/远程工具分层：文件读写、截图、本地自动化在本地；LLM 推理、RAG、业务系统调用在后端。

### 2.5 "个人/企业知识库" —— ✅ 需要，但用同一套基建（第 7 节）
- 需要，但**不要建两套系统**。用 `scope` 元数据维度区分 `personal:<uid>` 与 `enterprise`，一次带过滤的查询同时覆盖两者。
- 重要简化：**个人 KB 若 < ~20 万 token，可直接塞进上下文、跳过 RAG**（Anthropic 阈值），省掉大量检索基建。

### 2.6 "FSE 上门建 skill，按职位隔离可见，同职位全公司共享" —— ⚠️ 核心创意对，但要平台化
- **对**：Salesforce 的 forward-deployed engineers、腾讯的服务商生态都证明"专业服务上门"是刚需。
- **风险 1（成本/规模）**：纯靠 FSE 人力堆，边际成本高、难规模化。必须配**低代码 skill 构建工具 + 场景模板库 + 服务商认证体系**，让 FSE 从"写代码"变成"配模板"，并逐步把能力下放给客户管理员与第三方服务商。
- **风险 2（技能库治理）**："同职位全公司共享 skill" 要当**软件供应链**管理：skill 是可执行代码/指令，可被投毒（tool poisoning）。需版本化、审核、签名、灰度发布，并用 `allowed-tools` 把每个 skill 绑定到其角色可用的最小工具集。
- **风险 3（"按职位可见"的实现）**：不要新建一套平行的"职位→skill 可见性"手工规则（会与权限系统漂移）。应把 skill 可见性也纳入统一授权图（OpenFGA）：`role → 可见 skill 集`，与数据权限同源。

**结论**：你的 first-draft 方案骨架正确（通用基座 + 职位化 skill + FSE 落地 + 权限隔离 + 知识库），**不是需要推翻，而是需要三处升级**：(a) skill 收敛为窄工具 + 确定性工作流兜底；(b) 权限从"静态两档"升级为"ReBAC + 从 ACL 涌现"；(c) FSE 从"人力交付"升级为"平台化交付 + 服务商生态"。

> 完整画像与来源见 01/02 分文件。这里只提炼"我们该抄什么、该避什么"。

### 3.1 国内竞品

| 产品 | 定位 | 权限模型 | 对我们的启示 |
|---|---|---|---|
| **腾讯乐享 Lexiang** | 企业知识库 + AI 问答（你的真实参照物）| **4 层隔离**（企业→团队→KB→文档），按部门/角色/个人配置；不同终端用户可绑不同 KB；system-bot 只能取公共知识 | **直接照抄它的分层权限 + 应用侧用户分级 KB**；bot 最小权限原则 |

| **飞书 aily / 扣子 Coze** | 飞书内企业 Agent 平台 / 通用 no-code Bot | aily 继承飞书 RBAC；Coze 企业版权限+OAuth scope | 扣子有 OSS 自托管版（真 on-prem）可参考 |
| **百度 文心/千帆、华为 盘古** | no-code agent / 行业模型 + agentic | 云 IAM 角色制；华为最强真 on-prem（Cloud Stack）| 华为=重交付/上门参照；百度千帆私有部署 |
| **RPA 数字员工（影刀/来也/实在/金智维）** | 数字员工，RPA+LLM | 厂商各异，常按 bot | **FSE 参照系**：驻场常态、私有部署标配、UI 自动化桥接无 API 老系统 |

**国内共性与关键发现**：
- 大平台都绑生态（企微/钉钉/飞书的组织架构 = 天然身份与权限源），强调 no/low-code 构建，**都不做官方驻场 FSE**，走服务商/partner + "客户成功团队" + **预打包垂直套件**（阿里 OPT 最锐利）。
- **权限两大成熟模式**：identity-passthrough（钉钉悟空）+ 平台层角色 scope 绑定（腾讯 ADP 的 `visitor_biz_id`）——我们两者都要支持（详见 6.3）。
- **真正的 FSE 参照是华为 + RPA/数字员工厂商**（驻场交付文化 + UI 自动化，详见 9.3）。

### 3.2 国外竞品（按权限成熟度排序）

**Glean ≈ Microsoft ≈ Amazon Q ≈ Google（都强，ACL-at-query-time）> ServiceNow（强但单平台）> Salesforce（最弱，权限执行甩给实施方）**

- **Glean（权限感知检索的标杆，重点学它）**：连接器同时抓"内容 + 权限图 + 身份"，三类必须暴露 Content/People/Activity；查询时用户身份+组与每篇文档 ACL 求交；配 DSPM 治理层查越权。
- **Microsoft Copilot Studio + M365 Copilot**：run-as-user，靠 Microsoft Graph 权限 + Purview 敏感度标签；敏感度标签会**流到生成结果**（继承最严格来源标签）。
- **Amazon Q Business**：**⚠️ 2026-07-31 对新客户关闭**（转 Amazon Quick），只能当参考。亮点是 Trusted Identity Propagation（IdP JWT → Identity Center token → identity-aware 凭证），但**只传 Identity Center 里的组**，SCIM 组没同步全就会静默漏配。
- **Google Agentspace / Gemini Enterprise**：Workforce Identity Federation 传 `google.subject`/`google.groups`，查询时过滤。
- **ServiceNow / Now Assist**：复用既有 ACL；亮点是**调用权与数据权分离**（`securityAcl` vs `dataAccess`）——谁能调 agent ≠ agent 能碰谁的数据。
- **Salesforce Agentforce（反面教材）**：agent 以自己的 "Agent User" 身份行动，端用户身份传递靠实施方自己搞；胖 agent 账号 = 跨角色泄漏风险；ForcedLeak（CVSS 9.4）实锤。

### 3.3 三条跨竞品铁律（务必内化）
1. **filter-before-grounding**：先按用户 ACL 过滤，再喂大模型。永不 post-filter 生成结果。
2. **ACL 随内容一起爬取/同步**，统一归一到内部身份 schema（锚定 IdP，SCIM 同步）。**组同步是头号静默失败点**——当一等大事对待。
3. **假定提示注入必然发生**：权限放数据层 + 出口控制（输出目的地白名单），绝不放提示词里。

### 3.4 市场冷水（来自 Agentforce 批判性复盘，见 02d）
- CRMArena-Pro：多轮任务成功率 ~35%；Salesforce 悄悄退回确定性控制（指令 >8 条模型就丢）。
- "拦截率 ≠ 解决率"；71% 支持负责人把"注水指标"列为信任 AI 厂商的头号障碍。
- Gartner：2027 前 >40% agentic 项目将被取消。**→ 我们靠"诚实指标 + 严格权限 + 落地服务"differentiate。**

### 4.2 核心组件职责

- **Orchestrator（编排层）**：不是"让 LLM 自由发挥"，而是**确定性工作流为骨、LLM 为肉**。每个场景 = 一个状态机（LangGraph 风格），节点内才调 LLM。每次工具调用前必过授权层。破坏性/对外动作走人在环。
- **授权层（PDP, Policy Decision Point）**：**OpenFGA** 为核心引擎。三重职责：(a) 粗粒度 RBAC 守门（这个角色能否用 manager-agent、能用哪批工具）；(b) 细粒度 ReBAC（这个用户对这条记录/这篇文档有无权限）；(c) ABAC 上下文条件（时段、数据驻留、会话范围）。**每次工具调用 + 每次检索都实时 Check。**
- **知识库/RAG 层**：Qdrant（Apache 2.0）单集合 + payload 分区；混合检索（BM25+向量）+ reranker；ACL 作为 chunk 元数据**服务端预过滤**；`scope` 维度分个人/企业。
- **工具/连接器层**：每个企业系统连接器 = 一个 **MCP Server**（OAuth 2.1 资源服务器）。工具可用性由授权层按角色 gate。远程连接器走 Streamable HTTP，本地能力（文件）走 stdio。
- **身份与数据同步层**：对接客户 IdP（Entra/Okta/企微/钉钉），SCIM 同步用户与**组**（组是关键！）；源系统 ACL 增量同步进 OpenFGA 关系图；维护权限新鲜度。
- **横切**：OpenTelemetry 可观测（Agentforce 3 已把这做成标配）；租户隔离；哈希链审计日志（满足合规与回溯）；模型路由支持 BYOLLM 与"推理位置分级"（敏感推理可留私有）。

### 4.3 一次请求的完整数据流（以"查订单进度"为例）

```
1. 销售在桌面 App 问"XX订单进度？"
2. App 附带用户 IdP 令牌 → 网关验证身份
3. Orchestrator 路由到"订单进度查询"场景模板(确定性工作流)
4. 工作流节点要调 [ERP-订单查询] MCP 工具
   → 授权层 Check：该用户+该角色 能否调此工具？(RBAC) 
   → Token Exchange：换出 act=用户/actor=agent 的短时令牌
5. MCP Server 以"用户身份"查 ERP，只返回该用户有权看的订单
6. 若需补充知识(如工艺说明) → RAG 检索
   → 服务端注入 ACL 预过滤：tenant_id + (enterprise∩用户授权 OR personal:uid)
   → 混合检索 + rerank → 只有授权 chunk 进入上下文
7. LLM 节点基于"已授权的数据"生成进度报告
8. 全过程写哈希链审计日志(谁、调了什么工具、什么参数、授权决策)
9. 返回销售；若涉及写操作(改订单)→ 先弹人在环确认
```

**这条链的安全保证**：第 4、5、6 步的授权都发生在**数据到达 LLM 之前**；agent 用的是用户身份的降权令牌，**不可能越权**；即使提示注入让 LLM "想"越权，工具层和检索层的 ACL 也会拦住。

## 5. 技术选型

### 5.1 选型总表

| 层 | 首选 | 备选 | 理由 / 关键约束 |
|---|---|---|---|
| **授权引擎** | **OpenFGA**（Apache 2.0, CNCF）| SpiceDB（强一致场景）/ Cerbos（纯属性）| Zanzibar 系；Auth0 FGA 底座；自托管简单；多租户原生；有 RAG/agent 授权一方文档 |
| **向量库** | **Qdrant**（Apache 2.0）| pgvector（已重度用 PG 且 <1000万向量）| 一流的过滤检索 + payload 分区多租户；ACL 预过滤刚需 |
| **Agent 编排** | **LangGraph**（MIT）| 微软 Agent Framework（MIT, .NET）| 确定性状态机骨架 + LLM 节点；可控、可测 |
| **RAG 框架** | **LlamaIndex**（MIT）| 自研薄封装 | 元数据过滤检索是权限 KB 的正确机制 |
| **桌面客户端** | **Tauri v2**（MIT/Apache）| Electron（纯 JS 团队）| deny-by-default 路径/能力沙箱最适合碰本地文件的 agent |
| **连接器协议** | **MCP**（Streamable HTTP + stdio）| 自定义 REST | 事实标准；大厂已采纳；本地+远程统一 |
| **身份** | OIDC + **SCIM** 同步 | — | 组同步是权限正确性的命脉 |
| **可观测** | **OpenTelemetry** | — | 行业标配（Agentforce 3 Command Center 即基于此）|
| **文档解析** | RAGFlow（Apache 2.0）作子系统 | Unstructured | 深度解析表格/PDF/版式 |

### 5.2 要不要基于开源平台改造？——关键结论

**结论：不要把 Dify/FastGPT/n8n 当"产品基座"，但可借鉴组件与参考实现。**

**许可证是第一道过滤器（会在功能之前淘汰最完整的平台）：**

- **❌ Dify & FastGPT**：改过的 Apache 2.0，**禁止未经书面商业授权运营多租户环境**。而我们的"给多家企业做 agent 平台"恰恰是多租户 SaaS——直接踩雷。且都锁品牌。
- **❌ n8n**：Sustainable Use License，仅限内部使用，不能拿去卖衍生产品。
- **⚠️ Open WebUI**：有最好的原生权限过滤 RAG 和免费 RBAC，但白标超过 50 用户需企业授权——好参考，授权是地雷。
- **⚠️ Flowise**：需要的 SSO/RBAC/workspace 在商业授权的 `enterprise/` 目录里。

**两个 MIT、可商用的最佳候选（按"哪个更难自建"来选）：**

1. **AnythingLLM（MIT）**：唯一已经现成具备我们最难需求的——**Windows 桌面 app + 本地文件读写**（File System Agent），还有 MCP 客户端、workspace 级文档隔离。你要在其上自建：组织/租户层级、细粒度 RBAC、按角色 skill 隔离、按用户的权限过滤 RAG。（注意：研究期间刚修过一个跨 workspace 越权 bug，隔离需独立审计。）
2. **LibreChat（MIT）**：免费能拿到的**最强权限模型**——每个实体（agent/prompt/MCP server/文件/会话）都带自己的 ACL，支持按用户/组/角色共享；MCP 工具按用户上下文注入。没有桌面 app（需自建）。**因为我们产品的承诺就是"严格权限隔离"，LibreChat 的 ACL 底座是更难复制的东西**——若"到桌面的时间"不是最紧的，倾向 LibreChat + 自建桌面客户端。

**"建得更对"的组合（推荐真正的自研路线）**：LlamaIndex（RAG，元数据过滤是权限 KB 的正确机制）+ LangGraph 或微软 Agent Framework（编排）+ RAGFlow（文档解析子系统）+ Goose（Apache 2.0，MCP 原生桌面参考实现）。

### 5.3 两个所有基座都缺、必须我们自建并审计的能力
1. **按角色/租户对工具与 skill 代码做沙箱**——没有任何候选现成做到。继承来的工具系统要当"shell 访问"级别对待。
2. **按用户权限过滤的检索（permission-filtered retrieval）**——只有 Open WebUI 原生有。这是我们的核心竞争力，必须自建 + 独立安全审计。

### 5.4 模型策略
- **多模型可插拔 + BYOLLM + 推理位置分级**是企业刚需（Salesforce/Agentforce 已印证）。架构从一开始就要支持：默认云端模型、可切私有/本地模型（Ollama 用于隐私/离线/断网）、敏感推理留在信任边界内。
- 构建 AI 能力时默认用最新最强的 Claude（Opus 4.8 / Sonnet 5 等），但保持模型层可替换。

## 6. 权限隔离方案（第一硬需求，详解）

### 6.1 分层授权模型（没有单一模型能扛住生产）

```
┌─────────────────────────────────────────────────────────┐
│ RBAC（粗粒度守门）                                          │
│  这个角色 → 给不给 manager-agent？能用哪批工具/skill？      │
├─────────────────────────────────────────────────────────┤
│ ReBAC（细粒度核心，OpenFGA）★ 重心在这里                   │
│  "这个团队的经理 → 能看这个团队的这些订单/文档"            │
│  权限沿关系向下继承，不横向/向上泄漏                        │
├─────────────────────────────────────────────────────────┤
│ ABAC / 条件（上下文）                                      │
│  时段、数据驻留地、会话范围、风险等级                       │
└─────────────────────────────────────────────────────────┘
```

这与行业共识及 WorkOS 的 "FGA = RBAC + 层级" 一致，也正是 Moveworks 归一 RBAC/ABAC/ReBAC 的做法。

### 6.2 三个必须同时做对的执行点

**(1) 工具级授权** —— 每次 skill/工具调用触发实时 OpenFGA `Check`
- OAuth scope 太粗，真正的"谁能调哪个工具触碰哪个资源"要过 FGA。
- **强制"交集"语义**：只有当 **用户 AND agent 都被授权**触碰该资源时才放行——这从结构上把 agent 能力**封顶在用户权限之内**。
- 加参数校验、升权时重新认证、敏感动作人在环。

**(2) 数据级授权（RAG）** —— filter-before-grounding
- 把源系统 ACL 同步进 FGA 关系图。
- **混合过滤**：粗粒度预过滤（`ListObjects` 拿到授权 ID 集 → 灌进向量库元数据过滤，且带 tenant 命名空间）+ 细粒度后过滤（`BatchCheck`，over-fetch 2–3 倍再筛）。
- 检查点在**检索之后、喂 LLM 之前**。同一套过滤也用于 agent 记忆。**绝不靠 LLM 做访问控制。**

**(3) 身份传递** —— agent 永不超过用户
- Agent 默认是"confused deputy"（被利用的代理，见 2025-07 的 ForcedLeak）。
- 用 **OAuth 2.0 Token Exchange（RFC 8693）委派**：组合令牌带 `act` claim（用户=subject，agent=actor），受众域限定、短时、可衰减。
- 提供方凭证放 **Token Vault / broker**，agent 不持有长期宽权限；向零常驻权限（zero-standing-privilege）+ 按任务 JIT 升权演进。

### 6.3 "manager-agent vs staff-agent" 具体怎么落

**核心洞察（来自 01 国内竞品综合）：manager-agent 与 staff-agent 应靠"角色绑定 + 身份"区分，而不是克隆两个不同的 agent。** 是同一个 agent，因调用者的身份与角色不同而表现不同。国内两大成熟模式都印证：
- **模式 A：identity-passthrough（身份透传）**——钉钉悟空最典型：agent 以员工身份行动，只能看/做该员工在钉钉里本就有权的事（"数据权限从人的维度隔离"），组织级数据需管理员审批、敏感动作需二次确认。飞书 aily 同样继承飞书 RBAC。
- **模式 B：平台层显式角色→知识 scope 绑定**——腾讯 ADP 最佳实践：应用侧 RBAC（定义角色→绑用户→权限并集），**用户身份作为 `visitor_biz_id` 附在每一次 API 调用上**，因此隔离即使透过发布后的 API 也被保留；还有 External Privilege System 把细粒度授权委派给企业自己的 OA 系统。

**我们应两者都支持**，落到前述分层：
1. **RBAC 层**决定"能不能用某类 agent 与某批工具"（如审批类工具只给 manager 角色）。
2. **ReBAC 层**决定"能看哪些具体数据"——经理因为是"团队 X 的 manager"这一**关系**，自动能看团队 X 的订单；员工只能看自己经手的。**同一个 agent、同一个问题，两人检索到的数据集因 ACL 不同而天然不同。**
3. **每次检索/工具调用都携带调用者身份**（对应 `visitor_biz_id`/Token Exchange 的 `act` claim），让隔离贯穿到连接器与发布后的 API。
4. 这样组织调整（换岗、升职）只需改关系图，不必重写规则，避免漂移。

### 6.4 Skill 可见性 = 也纳入同一张授权图
- "同职位全公司共享 skill"实现为：`role:销售 → 可见 skill:{订单查询, 客户360, ...}` 的关系元组。
- **不要**建平行的手工可见性表——与数据权限同源于 OpenFGA，才不会漂移。

### 6.5 多租户隔离
- **默认**：共享 FGA store，用一个 `tenant` 类型建模边界；向量库按租户命名空间。
- **强合规租户**：store-per-tenant / index-per-tenant 物理隔离。
- 按租户设速率/花费上限，防"吵闹邻居"。

### 6.6 审计（合规刚需）
- 每个动作一条**防篡改、哈希链**事件：输入、推理、工具+参数、授权身份 + 委派链、租户、授权决策结果。
- 满足 SOC 2 Type II 与 EU AI Act 可追溯性；也是对抗"指标注水"信任危机的差异化。

### 6.7 组同步——头号静默失败点（务必重视）
- Amazon Q、Glean、Google 的共同教训：**组成员不全/过期会导致 ACL 静默漏配或误配**。
- 对策：SCIM 全量同步用户+组；独立于内容的**权限定时重同步**（权限比内容变得勤）+ 事件驱动更新；处理源系统限流；`acl_version` 追踪。
- **设计成 fail-closed（配错就拒绝），而非 Amazon/Salesforce 那种 fail-open 静默放行。**

## 7. 知识库 / RAG 方案

### 7.1 核心决策
- **向量库 Qdrant**（Apache 2.0）：单集合 + payload 分区做多租户（官方推荐，不要一租户一集合）；payload index 建在 tenant 字段。已重度用 Postgres 且 <1000 万向量则 pgvector（可蹭 Postgres RLS 做硬隔离）。
- **检索 = 混合 + rerank**：BM25（词法，抓 SKU/订单号/工艺代号等精确项）+ 稠密向量 + reranker。纯向量不再是合格基线。
- **ACL 作为 chunk 元数据**：摄取时把 `allowed_users + allowed_groups + deny` 写进元数据，服务端从已认证主体注入过滤器，**绝不信客户端传来的过滤条件**。

### 7.2 个人 KB vs 企业 KB —— 同一套基建，用 scope 区分
- 每个 chunk 带 `scope` 维度：`personal:<uid>` 或 `enterprise`（+ 更细的企业子 scope）。
- 检索时一次查询同时覆盖：`(scope==enterprise AND 用户已授权) OR (scope==personal:<当前用户>)`，合并 rerank。**不是两套系统。**
- **重要简化**：个人 KB 常常很小——若某用户语料 < ~20 万 token，**直接塞进上下文、跳过 RAG**（Anthropic 阈值），按用户决定，省大量基建。

### 7.3 摄取与同步（连接器同时抓内容 + ACL）
- **连接器 per 源**：内容与 ACL 一起拉（Glean/Amazon Q 模型）。**永不索引没有 ACL 的内容。**
- 增量索引（CDC/delta），非全量重爬；`source_doc_id + updated_at + acl_version` 追踪。
- 权限独立于内容定时重同步 + webhook 事件更新。

### 7.4 GraphRAG——暂缓
- 先上"混合向量 RAG + 上下文检索（contextual retrieval）+ rerank"。
- 只有当出现真实的"跨全库多跳/全局性"问题时，再考虑 GraphRAG/LightRAG——它索引成本高、运维重。
- 上下文检索（Anthropic）性价比高：给每个 chunk 预置 50–100 token 的定位上下文，配 prompt caching 约 $1.02/百万文档 token，失败率降约 35–49%。

### 7.5 增强/兜底
- 冲突知识（同一问题多篇矛盾文档）会被误当幻觉——Salesforce Help 上线时就踩过。做好知识治理与来源标注。
- 检索结果带引用（citation）以降幻觉、可核验。

## 8. Windows 桌面客户端方案

### 8.1 框架：Tauri v2 优先
- **deny-by-default 的能力/权限/路径 scope 系统**——精确限定 WebView 能调哪些命令、命令能碰哪些路径。对"有本地爆炸半径"的 agent 是最有价值的属性。
- 体积小（安装包 ~2–10MB，内存 ~20–50MB）、Rust 内存安全。
- **退路 Electron**：团队纯 JS、或需跨异构机型像素级一致渲染时。代价：自己扛 Chromium CVE 补丁（2026 有 context-isolation 绕过 CVE，负担真实）。
- WinUI 3/WPF 仅在全微软栈诉求时（换 MSAL/WAM 免费鉴权 + 原生 UI Automation）。

### 8.2 本地 vs 远程工具切分（瘦客户端 + 云端大脑）
- **本地跑**：受 scope 限制的文件读写、本地自动化、截图——放在受信核心里的类型化 broker 后，强制路径 scope + 用户同意。
- **远程跑**：LLM 推理、RAG、需服务密钥与中央策略/审计的业务工具。
- 两边都用 **MCP** 表达：本地 stdio server（仿 Claude Desktop，可用 MCPB/`.mcpb` 打包内置 Node 运行时）+ 后端 Streamable HTTP server。
- **优先结构化 UI Automation/RPA（FlaUI）而非截图坐标式 computer use**；纯视觉 computer use 只留给无 API 的应用，且必须沙箱化 + 最小权限 + 人工监督。
- **Ollama** 不做主大脑，但作隐私/断网/离线档与本地预处理（PII 脱敏）有价值。

### 8.3 鉴权
- OAuth 2.0 **Authorization Code + PKCE**，走**系统浏览器 + loopback 回调**（RFC 8252；绝不用嵌入式 webview）。App 当 public client。
- 微软栈用 **MSAL + WAM broker**。
- refresh token 存 **Windows Credential Manager（DPAPI 支撑）**，access token 只在内存；断网/无头场景提供 device code flow。

### 8.4 部署与管理
- **MSIX via Intune / App Installer**（现代托管机群）；**MSI**（GPO/SCCM 老环境）。
- **OV 证书签名**（EV 已不再买到即时 SmartScreen 信誉，2024 起信誉按版本累积）。
- 自动更新：`.appinstaller`（MSIX）/ electron-updater / Tauri updater，指向内网 HTTPS/UNC（支持断网/隔离网）。

### 8.5 安全
- 最小权限 scope；破坏性/高影响动作（删除/覆盖/批量移动/有后果的点击）**人在环 + dry-run + 默认可撤销**。
- 把所有文件/网页/截图内容当**不可信输入**（提示注入隔离）；出口白名单防外泄；本地防篡改审计日志上报 SIEM。

## 9. MCP + Skills 体系与 FSE 交付模式（你的核心创意落地）

### 9.1 MCP：企业连接器的标准
- MCP 架构：hosts / clients / servers；servers 暴露 tools / resources / prompts。传输：stdio（本地）、Streamable HTTP（远程）。
- **鉴权真相**：MCP server 只是 **OAuth 2.1 资源服务器**——它验证由独立授权服务器（你的企业 IdP）签发的令牌并校验 scope，**自己不签发令牌**。未授权调用返回 401 指向 Protected Resource Metadata（RFC 9728），客户端跑 OAuth+PKCE，再带 Bearer 令牌重发，server 必须校验令牌**受众**（RFC 8707）。
- **RBAC 要我们自己建**：MCP 给机制，我们做"scope → 角色 → 允许的工具"映射，最小权限渐进式 scope（用 `WWW-Authenticate` 挑战升权），并按角色过滤工具列表。
- **企业主流模式 = MCP 网关**：做 RFC 8693 token exchange（on-behalf-of），把宽用户令牌降权成每 server 的窄令牌，跨跳传播用户身份（subject=用户，actor=agent）做干净审计；对遗留 PAT/API-key 后端用 Vault 兜底。

### 9.2 Skills：角色化、公司共享的技能库
- 一个 Skill = 一个文件夹，含 `SKILL.md`（YAML `name`+`description` + Markdown 正文 + 可选脚本），**渐进式披露**加载（元数据常驻；正文相关时加载；文件按需）——所以别的角色用不到的 skill 几乎零成本。
- Skill 可分级：企业 / 个人 / 项目 / 插件级。
- **干净的映射**：一个 skill 对应一个"角色-任务"，打包成**按角色的 plugin**（skills + MCP 配置 + hooks），版本化，经私有 marketplace（内部 Git 仓）分发；**按角色可见 = 安装对应角色 plugin + 企业级 scope 管控**。
- **`allowed-tools` frontmatter** 把每个 skill 绑定到其角色可用的最小 MCP 工具集——与 9.1 的 scope 强制正好咬合。
- **Skills 教"怎么做"（指令/流程知识），MCP 连"能做什么"（能力）**，互补（Anthropic 的定位）。

### 9.3 你的 FSE 模式——如何做对（升级版）

你的原始想法："运维（FSE）上门帮企业建 skill，按职位隔离可见，同职位全公司共享。"

**保留的对的内核**：企业 Agent 落地是重服务，上门/驻场是刚需（Salesforce forward-deployed engineers、腾讯服务商生态都印证）。

**三层升级**：

1. **把 FSE 从"写代码"变成"配模板"**
   - 建**场景模板库**（订单进度、生产计划、物料齐套、质检报告……）：每个模板 = 确定性工作流 + skill 绑定 + 提示词 + 权限声明。
   - FSE 上门主要做：接连接器（MCP Server 连 ERP/MES/CRM）、选模板、映射该企业的角色→权限→skill 可见性。腾讯 ADP 的时间线可参照（FAQ 1–2 周、多场景 4–6 周）。

2. **把 skill 库当软件供应链治理**
   - skill 是可执行指令/代码，有 **tool poisoning / rug-pull** 风险（描述里藏恶意指令）。对策：把工具描述当**不可信数据**；skill **版本化 + 审核 + 签名 + 灰度发布**；FSE 编写的 skill 像依赖一样审计，执行沙箱化。
   - "同职位共享"= 经审核的 skill 发布到该角色的 plugin，全公司该角色自动获得。

3. **把"按职位可见"纳入统一授权图**
   - `role → 可见 skill 集` 用 OpenFGA 元组表达，与数据权限同源，杜绝平行手工表漂移。

**商业模式含义**：
- FSE 是**获客与留存的护城河**（重服务壁垒），但边际成本高。路径：**自营 FSE 打样 → 沉淀模板与工具 → 培训认证第三方服务商 → 能力逐步下放给客户管理员自助**。这是把"人力交付"转为"平台化交付 + 生态"的必经之路。
- 一个务实预期：初期像 Palantir/Salesforce 一样"重"，靠 FSE 把前几十家做深、沉淀可复用资产，再逐步轻量化。

**国内的 FSE 参照对象要选对（来自 01 深调）**：
- **大平台（腾讯/阿里/字节/百度）都不做官方驻场 FSE**——反而是 self-serve no-code + **预打包垂直套件**（阿里 OPT "单人团队" 是最锐利的例子：为 10 个行业打包 skill+工作流+数据）+ 服务商生态。**这提示：模板/套件化是降 FSE 成本的关键武器。**
- **真正该参照的是两类**：(a) **华为**——重交付 + 上门工程 + 私有部署（继承其政企交付传统，是大厂里最接近 FSE 模式的）；(b) **RPA/数字员工厂商（影刀/来也/实在/金智维）**——**驻场交付是常态**（继承 RPA 交付文化）、**私有部署是标配卖点**（数据主权）、且用 **UI 自动化桥接无 API 的遗留系统**。
- **对你的直接启示**：你的"FSE 上门 + 桌面 agent 能读写本地/做自动化"设想，本质是"RPA 数字员工的交付文化 + LLM 的理解能力"的结合。**当无 API 的老系统挡路时，桌面端的 UI 自动化（第 8.2 节的 FlaUI 路线）就是 FSE 的兜底武器**——这正是 RPA 厂商的看家本领，值得吸收。

### 9.4 治理缺口提醒
- 组织级"全公司 skill 治理/分发/管控"在 claude.ai 消费端是缺失的——企业级分发要跑在 **Claude Code / Agent SDK / 自建 host** 上才能拿到管理与分发控制。我们本来就要自建 host，正好覆盖。

## 10. 分阶段路线图（后端优先）

### Phase 0 — 地基与验证（约 4–6 周）
- 搭编排层骨架（LangGraph）+ 一个确定性场景（"查订单进度"）跑通。
- OpenFGA 起步：建租户/角色/关系模型；工具级 Check 打通。
- 一个 MCP Server（连一个真实/模拟 ERP），走 Token Exchange 身份传递。
- 最小可用审计日志。
- **验收**：销售问订单进度，agent 以用户身份查 ERP，越权数据取不到，全程有审计。

### Phase 1 — 权限感知知识库（约 6–8 周）
- Qdrant + 混合检索 + rerank；ACL 元数据 + 服务端预过滤。
- 连接器同步内容 + ACL；SCIM 同步用户/组；权限重同步机制。
- 个人/企业 scope；小个人 KB 走"直接入上下文"简化。
- **验收**：同一问题，经理与员工检索到不同数据集；组变更后权限正确更新。

### Phase 2 — Skill 体系与 FSE 工具链（约 8–10 周）
- Skill 打包/版本化/签名/灰度；`allowed-tools` 绑定；角色可见性入 OpenFGA。
- 场景模板库（3–5 个高频生产场景）+ FSE 配置化上线流程。
- MCP 网关（token exchange 降权、审计、Vault 兜底遗留系统）。
- **验收**：FSE 用模板 + 配连接器，1–2 周内给一家新企业上线一个职位的 agent。

### Phase 3 — Windows 桌面客户端（约 8–10 周，可与 P2 并行起步）
- Tauri v2 客户端：聊天 UI + PKCE 登录 + 本地文件 MCP（路径沙箱）。
- 本地/远程工具分层；破坏性操作人在环 + dry-run + 撤销。
- MSIX/MSI 打包、OV 签名、内网自动更新。
- **验收**：桌面 app 能读写受限本地目录、调后端 agent、破坏性操作被拦截确认。

### Phase 4 — 企业级硬化与规模化（持续）
- 多租户物理隔离档、可观测（OpenTelemetry）、DSPM 越权检测层。
- BYOLLM / 私有模型 / 推理位置分级。
- 服务商认证体系；客户管理员自助能力下放。
- 合规（SOC 2 / 等保 / EU AI Act 可追溯）。

> 时间为粗估，取决于团队规模与并行度。P0/P1 是不可跳的地基；P2/P3 可部分并行。

## 11. 风险清单与对策

| 风险 | 严重度 | 对策 |
|---|---|---|
| **权限越权/数据泄漏**（跨角色、跨租户）| 🔴 极高 | filter-before-grounding；agent 权限≤用户；OpenFGA 实时 Check；fail-closed；独立安全审计 |
| **提示注入 → 越权/外泄**（ForcedLeak 类）| 🔴 极高 | 权限放数据层非提示词；外部内容当不可信；出口白名单；工具参数校验 |
| **组同步不全 → ACL 静默漏配** | 🔴 高 | SCIM 全量同步用户+组；权限独立重同步；fail-closed；acl_version 追踪 |
| **Skill 供应链投毒**（tool poisoning）| 🟠 高 | skill 版本化+审核+签名+灰度；工具描述当不可信；执行沙箱 |
| **自主 agent 不可靠**（多轮成功率低）| 🟠 高 | 确定性工作流骨架；窄 skill；关键路径人在环；诚实指标 |
| **本地破坏性操作**（删/覆盖文件）| 🟠 高 | 路径沙箱；人在环；dry-run；默认可撤销；审计 |
| **开源基座许可证陷阱**（多租户/白标）| 🟠 中高 | 避开 Dify/FastGPT/n8n 作基座；选 MIT/Apache；许可证独立法务复核 |
| **权限新鲜度滞后**（镜像快照）| 🟡 中 | 事件驱动 + 定时重同步；高敏场景走 live check |
| **FSE 交付不规模化**（人力堆）| 🟡 中 | 模板库 + 配置化 + 服务商生态 + 能力下放 |
| **市场泡沫/项目被砍**（Gartner >40%）| 🟡 中 | 聚焦可核验 ROI 的窄高频场景；诚实指标；快速见效 |
| **供应商锁定**（模型/云）| 🟡 中 | BYOLLM；模型层可插拔；连接器标准化(MCP) |

## 12. 待你决策的关键问题

在进入实现前，以下决策会显著影响架构，建议尽快明确：

1. **部署形态**：目标客户主要要 SaaS 多租户，还是私有化/本地部署？（影响租户隔离强度、许可证选择、交付成本）
2. **首批目标行业与场景**：先聚焦哪个行业（如离散制造/贸易）的哪 3–5 个高频场景？（决定场景模板库的起点）
3. **身份源**：客户主要用什么 IdP？（企微/钉钉？还是 Entra/Okta？决定身份传递与 SCIM 集成的优先级）
4. **自研 vs 基于开源**：倾向从零自研（LangGraph+LlamaIndex+OpenFGA+Qdrant 自组），还是在 LibreChat/AnythingLLM 上改造起步？（影响启动速度 vs 长期可控性）
5. **FSE 规模与商业模式**：初期自营 FSE 打样几家，还是一开始就要服务商生态？定价按席位/按用量/按项目？
6. **合规目标**：目标客户对等保/SOC 2/数据出境有何硬要求？（影响部署与模型选择）
7. **桌面 App 优先级**：桌面客户端是 P0 必需，还是可先做 Web + 企微/钉钉机器人验证价值、桌面后置？
