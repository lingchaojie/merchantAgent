# 补充：企业 Agent 创业公司深度画像（Writer / Sana / Moveworks）

> 说明：本文件是"02 国外竞品"调研的子代理产出。子代理回传父代理失败，故单独落盘保存。数据截至 2026 年 7 月。每家最深入的是"权限与安全模型"一节。

**重要前提更正：**这三家"创业公司"如今两家已被大厂收购。**Writer** 仍独立；**Sana** 被 **Workday** 收购（约 11 亿美元，2025-11-04 完成），现为 "Workday Sana"；**Moveworks** 被 **ServiceNow** 收购（约 28.5 亿美元，2025-12-15 完成）。

---

## 1) WRITER (writer.com)

### 价值主张与 Agent 构建方式
全栈企业生成式 AI 公司，**自研模型（Palmyra 系列，2020 起自研）+ 应用/Agent 层都做**。
- **Palmyra X5** —— 旗舰企业 LLM：约 100 万 token 上下文、自适应/混合推理、内建工具调用；2025 年 4 月发布，也上了 Amazon Bedrock。是 Writer agents 的推理引擎。
- **AI HQ**（2025-04-10）—— 构建、激活、监管 agent 的中枢，让 IT 与业务用户协作。
- **Agent Builder** —— 低代码/无代码可视化编辑器，拖拽式积木；业务用户设计 UI/提示词，开发者加高级逻辑。
- **WRITER Agent**（基于 Palmyra X5 的自主工作流 agent）与 **AI Studio** 开发者平台。

### 集成/连接器模型
- **WRITER Knowledge Graph** = 基于图的 RAG。连接器：Confluence、SharePoint、Google Drive、Notion、网页域名/页面、手动上传。
- 内容是**摄取/索引**（非纯实时）：连接器同步保持图更新；用"检索感知压缩"给带元数据的分块建索引。
- 近期加了 MCP 连接器。

### 权限与安全模型（最关键发现）
Writer 的访问控制是**角色/图范围的，不是源系统 ACL 传递的**—— 这是相对 Moveworks/Sana 最大的差异点。
- **组织 + 团队范围的 RBAC。**8 个默认角色（组织管理员、AI Studio 完全访问/构建者/团队管理员/只读、业务管理员、IT 管理员、团队管理员、团队成员）。多角色权限**叠加**。自定义角色仅企业版（最多 100 个）。
- **知识图访问在图/agent 级 + 数据分段级把关** —— Writer 自己的建议就是"为不同主题或部门建独立知识图"。团队所有成员都能查看团队 KG（不可按文档配置）。
- **⚠ 关键缺口/越权风险：**未找到任何公开文档证明知识图在查询时强制执行**按用户的源系统文档 ACL**（SharePoint/Drive/Confluence）。连接器鉴权用 OAuth/Microsoft Graph **应用级**权限，Writer 也说托管应用连接器"用预设权限、不可修改"，无细粒度按文档控制。含义：**任何能访问某 KG/agent 的用户，都可能检索到其在源系统本无权看的内容。**官方给的缓解是组织级分段（建独立 KG），而非文档级安全裁剪。
- **身份/开通：**SCIM/IdP 自动分配角色（IdP 为准）；管理审计日志记录角色变更。
- **认证：**ISO/IEC 27001、27701（隐私）、**42001（负责任 AI —— 少见）**；年度 SOC 2 Type II（含 HIPAA/HITECH）。
- **不用客户数据训练：**Writer 声明，斯坦福 CRFM FMTI 与 Google Cloud 佐证。
- **⚠ 重大事件（2026 年 7 月）：**已修复的**严重跨租户会话隔离漏洞**（组件 "WriteOut"，Sand Security 披露）—— agent 预览可把会话 cookie 转发给攻击者控制的 agent，导致会话 token 跨租户泄漏。虽已修复，但直接动摇了租户隔离的可信度。

### 优劣势
- **优势：**真正全栈（模型 + 应用层）；认证强（ISO 42001 罕见）；可信的不训练承诺；100 万 token 上下文；图 RAG；易用低代码构建器。
- **劣势：****权限模型粗**—— 角色/图级，无源文档 ACL 继承/查询时裁剪证据，KG 内越权风险真实、仅靠人工分段缓解；2026 年 7 月跨租户 token 泄漏漏洞（已修）是隔离污点；旗舰模型本地/私有部署有限；查询时授权透明度不如 Moveworks。

## 2) SANA AI → "Workday Sana"

### ⚠ 关键背景更正
Sana（Sana Labs AB，斯德哥尔摩）**被 Workday 收购** —— 2025-09-16 宣布，约 **11 亿美元**，**2025-11-04 完成**，现为 **"Workday Sana"**。

### 价值主张与 Agent 构建方式
职场 AI 平台，融合**企业搜索 + 助手 + 会议（记录）**，以公司知识为根基；源自**企业学习（Sana Learn / L&D）**。
- **Sana Agents** —— 无代码 agent 构建器；agent 自动化任务并主动行动：理解公司政策、调 API、更新记录系统、触发审批、跨 Workday/ITSM/CRM/协作工具完成端到端工作流。

### 集成/连接器模型
集成访问、读取并**索引**外部数据以支撑有依据的回答；数据**与源保持同步**，更新快速反映（实时/近实时）。

### 权限与安全模型
- **权限镜像（ACL 感知声明）：**访问权可**镜像所连外部系统（如 Google Drive、SharePoint）中设定的权限**，检索遵循这些外部权限，敏感数据仅限授权用户。作为能力描述，但**未公开检索时执行机制**（前/后过滤、ACL 刷新频率）。
- **原生层级模型：**工作区/文件夹/资产/Agent 级 × 所有者/编辑者/查看者。私有文件夹/文件非共享不可见；即便所有者"未经明确共享也无法访问某用户的私有文件夹、文件或提示词"。
- **Agent 风险缓解 —— 3 层**（模型级检测；平台级权限边界/护栏；写操作/外部动作的人在环）：
  - **提示注入：**持续监控/测试模型，agent 遇到提示注入攻击时**通知用户**。
  - **管理员护栏：**管理员按用户风险画像控制可用集成；**仅管理员可配置远程 MCP 服务器**（限制只读 vs 读写、选择哪些用户/组连接）；管理员控制允许的邮件/日历域名，被拦请求**在平台级阻断**。
  - **人在环：**所有执行写操作或与外部方交互的 agent 工具（发邮件、更新 CRM）都要**确认**；中间推理/工具计划可供审查。
- **认证/SSO：**免密 SSO；SAML 2.0 及所有 IdP。（SCIM **未公开确认**。）
- **租户隔离：**宣传为**单租户**部署。
- **基础设施与加密：**跑在 **Google Cloud**；静态 **AES-128 和 AES-256**，传输 **TLS 1.2**；内部访问最小权限 + RBAC；年度第三方渗透测试。
- **认证与数据使用：ISO 27001**、**SOC 2**（五项信任原则全覆盖）、**GDPR**；**不用客户数据训练。**
- **缺口：**检索时授权（前/后过滤、权限新鲜度）无独立/技术细节；**数据驻留未明确**；SCIM 未确认。

### 优劣势
- **优势：**明确的**源权限镜像**（真正的 ACL 感知声明）；**单租户**隔离；强 **agent 安全**模型（写操作人在环、管理员集成/MCP/域名护栏、提示注入告警）；不训练；认证齐；Workday 分发渠道。
- **劣势：**检索时执行与 ACL 新鲜度的**公开细节薄**（无法独立验证）；SCIM 与驻留未确认；RAG 架构公开信息不足；被 Workday 收购后产品身份处于变动期。

## 3) MOVEWORKS（现为 ServiceNow 旗下）

### 收购背景
ServiceNow 于 2025-03-10 宣布收购（约 **28.5 亿美元**，其史上最大），**2025-12-15 完成**；2026-02-26 ServiceNow 推出 "Autonomous Workforce"，将 Moveworks 并入 ServiceNow AI Platform。

### 价值主张与 Agent 构建方式
面向员工的**智能体式 AI 助手**，用于 IT/HR 支持与全企业自动化。
- **Agentic Reasoning Engine** —— 规划多步动作、选择/执行下一步、观察并自适应。
- **Agent Studio / AI Agent Builder** —— 构建连接系统、执行策略、编排工作流的 agent；"Agent Architect" 推荐架构。
- **Plugins** = 助手调用的工具；助手负责推理/编排，构建者写工具。
- 2025-11-05 "Dynamic AI Platform" 让团队按部门组装/部署/治理专用助手。

### 集成/连接器模型 —— 两种模式
1. **镜像（Mirroring）** —— 通过 API 把数据 + 权限复制进优化索引（"Instadex"）。
2. **实时 API（Live API）** —— 查询时以用户身份调源系统的搜索 API。
平台"无论权限如何实现"都能工作。

### 权限与安全模型（三家中最严谨）
这是 Moveworks 公开技术深度明显领先之处。
- **统一授权模型：**把 RBAC/ABAC/ReBAC 归一为共同的**"通用元组结构"**（对象与用户间的关系），明确借鉴 **Google Zanzibar、OpenFGA、SpiceDB**。核心问题：**"用户 X 对记录 Z 是否有 Y 访问权？"**在存储层或其上强制执行。
- **索引数据的四种策略：**镜像权限（ReBAC）、托管权限（ABAC）、对所有人公开、对所有人拒绝。
- **镜像/ReBAC：**复制源规则（用户/组/角色），**扁平化**为直接的 用户→对象 关系以降低组维护负担，存于 **OpenFGA**（ServiceNow 侧用 Redis），传播到内容索引。增量摄取用 webhook 或约 15 分钟轮询。
- **ABAC：**用 DSL 写用户属性（部门/国家）+ 对象属性规则，从集成导入，故不依赖第三方系统。适合渐进上线。同一集成不能同时跑 ABAC 和镜像。
- **实时检查（"Airlock" / 用户同意）：**实时搜索时，按用户 **OAuth token 原生强制权限** —— "实时安全查询外部系统以评估权限"。
- **双层执行（防越权）：**索引/存储层**预过滤**（OpenFGA 中的 ReBAC 规则）保证只有允许的内容离开索引，再在检索时**后过滤**（ABAC + 实时检查）。用户"只能看到明确授权的数据"。
- **Content Gateway（ReBAC 解析）：**客户网关暴露文件权限、组成员（仅直接成员，嵌套链由 Moveworks 自己跟）、用户列表。Moveworks **"在查询时解析特定用户能否查看特定文件"**。公开访问需刻意用通配符，非 VIEW 动作被 schema 拒绝 —— 降低误开放。
- **身份传递：**大量原生身份集成 + 自定义身份网关；外部内容受**用户 OAuth token + 显式同意**保护。
- **权限新鲜度：**离线流式管道**近实时**更新权限；Moveworks 自己也指出这很难（个人文档权限频繁变、部分系统缺增量更新 API）。
- **治理/审计：**集中式决策是可追踪的接合点，产出**详细审计日志**。
- **合规/部署安全：FedRAMP Ready → FedRAMP Moderate 授权**；On-Prem Agent 支持防火墙内系统；区域数据中心。
- **缺口：**"不训练客户数据"及正式 SOC 2 Type II / 租户隔离架构声明公开页不可直接引用（信任门户受限）。

### 优劣势
- **优势：****最严谨、文档最全的权限模型** —— Zanzibar/OpenFGA 式 ReBAC + ABAC + 实时 OAuth 检查，预过滤 + 后过滤，真正查询时解析"用户 X 能否看文件 Z"，镜像与实时两种模式都尊重源 ACL，近实时权限同步，FedRAMP Moderate，可 on-prem，强审计/治理。
- **劣势：**镜像带来**权限新鲜度滞后**（官方承认）；ReBAC 组扁平化增加摄取复杂度；不训练/SOC 2/租户隔离声明难从公开页验证；并入 ServiceNow 后集成/路线不确定。

## 跨厂商结论（针对核心问题：权限）
- **Moveworks** —— 最强也最透明：真正**查询时、源 ACL 感知**的授权（ReBAC/ABAC/实时 OAuth，前 + 后过滤）。"用户只能看到被授权内容"是硬需求时最佳参照。
- **Sana（Workday Sana）** —— **声称源权限镜像**（Drive/SharePoint）+ 强 agent 安全/人在环 + 单租户，但检索时执行机制公开少；ACL 感知按"厂商声明"看待，未独立验证。
- **Writer** —— **数据授权最弱**：角色/图范围访问，**无按文档源 ACL 继承证据**；越权靠人工分段 KG 缓解；叠加 2026 年 7 月（已修）跨租户会话 token 漏洞。模型自研、认证（ISO 42001）、不训练是其强项。

> 不确定性：这些厂商公开的查询时授权细节远少于超大厂。采购前对 Sana/Writer 的检索时执行、Moveworks 的不训练/SOC 2、三家的数据驻留都应向厂商直接确认。
