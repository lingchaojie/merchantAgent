## 1. 一句话架构

**Windows 桌面 app 是主产品**（agent 主要在这里用：本地文件、富交互、长报告、接企业系统办事）；**企微主要做鉴权与组织来源**——直接复用员工在企微里的部门/主管/标签划分权限，企业**无需再全量导入员工**；企微上的轻量问答只是**加分的第二渠道**。两个表面共享**同一后端大脑**（LangGraph 编排 + OpenFGA 权限 + Qdrant RAG + MCP 连接器），权限自企微组织架构**自动映射进 OpenFGA**，按"读+聚合+预警"的窄高频场景先落地，写操作后置强审批。

> **纠正（相对早期草稿）**：企微不是"中心"，而是"鉴权来源 + 加分渠道"；主战场在桌面 app。下文据此调整。

## 2. 企微的定位：鉴权与组织来源（主）+ 轻量问答渠道（加分）

### 2.1 应用模型：第三方应用为主 + 代开发为辅
- **第三方应用（主）**：一套 `suite_id` 服务所有租户，上架应用市场，每个授权企业得自己的 `corpid` + `permanent_code`。这是多租户 SaaS 唯一可扩展的模型，也承载按席位计费。
- **代开发应用（辅）**：给大客户/需专属实例或更深通讯录权限的客户。
- **自建应用**：不适用（单企业），仅内部自测用。

### 2.2 服务商授权链（端点已验证，详见 08）
```
收 suite_ticket(每~10min) → get_suite_token → get_pre_auth_code
 → 管理员在 3rdapp/install 授权 → 拿临时 auth_code → get_permanent_code
 → 存 {corpid, permanent_code, agentid} → get_corp_token(每租户) → 调企业接口
```
所有存储**按 corpid 分键**——这就是租户隔离的物理基础。

### 2.3 身份锚点：`open_userid` + `corpid`
- `open_userid` = 服务商全局唯一、跨该服务商多应用稳定 → 作为**会话身份与 OpenFGA subject 的规范主键**。
- 每租户 `userid` 可能是密文且仅在 corpid 内有意义。
- **PII（手机/邮箱/头像）2022 起需成员 OAuth 授权**——不要依赖 PII，一律基于 `open_userid` + 部门 + 标签建模。

### 2.4 企微组织架构 → OpenFGA（这是接企微的核心目的）
**你的核心诉求（本轮确认）**：复用员工在企微里既有的角色/职位来划分权限，企业上线时**不必再全量导入员工**。企微把"谁是谁的上级、谁在哪个部门"现成给了我们，直接映射：
- 租户 = `tenant:<corpid>`；用户 = `user:<open_userid>`
- 部门 → 带 `parent` 继承的层级对象；`is_leader_in_dept` → `manager` 关系
- 标签(标签) → 角色组；管理员来自授权信息/管理员回调
- **同步时机**：授权成功时全量 + 通讯录变更回调增量（离职 status=5、部门删除要删元组）；同步做成幂等 + 定期全量兜底对账。

**一个必须注意的现实**：企微里**可靠的结构化信号是"部门树 + 主管标记 + 标签"**；而"职位（position）"字段是**自由文本**，各企业填法五花八门（"销售""业务一部经理""外贸-张三"），**不能直接当枚举角色用**。因此：
- **角色主要从 部门 + 主管标记 + 标签 推导**（这些是结构化的）。
- **职位自由文本**由 FSE/管理员在开通时**一次性映射**到平台标准角色（如把含"经理/主管"的映射为 manager 档），或让管理员在后台确认。
- 角色定义落在**我们平台侧**（OpenFGA），以企微组织结构**播种**，之后可微调——既复用企微、又不被其自由文本字段绑死。

**这意味着**："manager-agent vs staff-agent"的区分**主要来自企微组织架构**，无需企业手工全量维护；组织调整跟着回调自动生效。这是接企微的**最大价值点**（不是渠道，而是"零导入的权限来源"）。

### 2.5 计费：与企微解耦，初期走体外合同（按席位）
- 决策 5 是"服务商生态 + 按席位定价"，但**按席位计费不必依赖企微**——我们在自己后端 + OpenFGA 里统计激活用户数即可按席位收费（自有合同/发票）。
- 企微原生的"接口许可（按账号）"是**可选项**，仅当走应用市场标准计费时才需要；初期**不建议**，体外合同更简单灵活。
- 结论：**计费不是接企微的理由**；企微只承担鉴权/组织来源（主）+ 轻问答渠道（加分）。

## 3. 主表面 = 桌面 App；企微问答 = 加分的第二渠道

**定位（本轮纠正）**：桌面 app 是主产品与主战场；企微问答类似"多开一个入口"，做简要问答、随手查、主动预警推送，**不承载重活**。

| 维度 | 桌面 App（**主**）| 企微问答（加分第二渠道）|
|---|---|---|
| 定位 | agent 主战场：深度工作、长报告、本地文件、办事 | 简要问答、随手查、预警推送、唤起桌面 |
| 输出 | 完整报告、表格、流式长文 | 短答/卡片（≤2048B）；长内容→"在桌面打开" |
| 本地能力 | 读写本地文件、UI 自动化(RPA 兜底)、截图 | 无 |
| 典型场景 | 生成报价单/PI、批量对账、跨系统聚合报告、改单前预览 | "齐套了吗""这单进度"随手问、交期预警推送 |
| 登录 | 企微扫码/OAuth → 换发本系统 JWT | 企微原生身份 |
| 连接 | 常规 HTTPS + 本地 MCP(stdio) | webhook 回调（公网 HTTPS + 备案）|
| 优先级 | **P0 就要**（主产品）| 后置/并行（加分项）|

**关键设计**：
1. **桌面优先**——agent 的主要能力（本地文件、长输出、跨系统办事）都在桌面；企微能做的是它的子集。
2. **身份统一**——桌面用企微扫码登录（`3rd_qrConnect` → `getuserinfo3rd` → 换发本系统 JWT，键为 `open_userid`+`corpid`），权限来自企微组织架构映射的 OpenFGA。**这是接企微的主要目的**。
3. **企微问答受限但够用**——企微智能机器人回复被限死在 2048B/3 分钟，所以它只做简要问答与预警；长内容用"卡片摘要 + 深链桌面"绕开限制。私有化版企微无智能机器人（但我们是 SaaS，且它只是加分渠道，无碍）。
4. **本地文件/富交互只在桌面**——企微侧不碰本地文件。

> **重要纠正（相对早期草稿）**：早期草稿把"企微机器人先行、桌面放 P2"是**错的**——那是把加分渠道当成了主表面。**正确次序：桌面 app 从 Phase 0 就是主交付物**；企微问答作为加分渠道可后置或并行（见第 7 节已按此调整）。

### 3.1 企微是否为唯一身份源？（建议）
你的诉求（复用员工既有组织架构、免全量导入）**不止企微能满足**——钉钉、飞书、AD/Entra 同理。建议把身份/组织接入抽象成**IdP 适配器接口**，**企微作为第一个实现**，后续按客户需要加钉钉/飞书/AD。成本很低（现在定好接口即可），却避免把产品绑死在企微上。若首批客户清一色用企微，实现上可以只先做企微，但**接口先留好**。→ 见第 8 节确认点。

## 4. 场景库与首批切片（贸易 + 制造）

### 4.1 完整场景库（26 个，详见 09）
- **贸易 12**：T1 客户历史成交价(高敏)｜T2 报价毛利+PI草稿(高敏)｜T3 订单进度与出货预警(中)｜T4 单证不符点校验(中)｜T5 客户信用/账期/应收(高敏)｜T6 可用库存+在途(中)｜T7 应收对账单(高敏)｜T8 物流轨迹(低-中)｜T9 供应商比价(高敏)｜T10 收汇核销(高敏)｜T11 老板销售驾驶舱(高敏)｜T12 询盘背景核验(中)。
- **制造 14**：M1 可承诺交期ATP(中)｜M2 齐套/欠料清单(中·超高频)｜M3 插单影响(中)｜M4 工单进度看板(低-中)｜M5 交期延误预警(中)｜M6 排产/重排(中-高·写)｜M7 IQC合格率(中)｜M8 IPQC/SPC(中)｜M9 供应商质量排名(高敏)｜M10 欠料比价催货(高敏·写)｜M11 订单实际成本(高敏)｜M12 设备稼动率(低-中)｜M13 厂长驾驶舱(高敏)｜M14 库龄/呆滞料(中)。

### 4.2 首批切片建议（按"价值高 × 风险低 × API 可读"三重排序）

**第一批（P0/P1，纯读+聚合+预警，企微机器人先行）：**
1. **M2 齐套/欠料清单**（制造超高频刚需，纯读，价值最高）
2. **T3 订单进度与出货预警**（你的原始场景，贸易/制造通用）
3. **M5 交期延误预警**（主动推送，企微卡片天然契合）
4. **T6 可用库存 + 在途核查**（跨系统聚合的典型）
5. **T5 客户账期/应收提醒**（高频，主动预警）

这批共同点：**只读、跨系统聚合、可主动推送、写操作为零**——风险最低、最快见效、最能证明"数据割裂聚合"的核心价值。

**第二批（P2，桌面 app + 轻写）：**
6. T2 报价毛利+PI 草稿、T7 应收对账单、M11 订单实际成本、M13/T11 驾驶舱——**长输出/富交互**，落在桌面；写=生成草稿存草稿箱（低风险写）。

**第三批（P3+，强审批写操作）：**
7. M6 排产重排、M10 欠料催货、改价/改单——**高敏/对外写**，强制人在环 + 审批 + 审计；无 API 老系统禁止 agent 直接 UI 写财务单据。

### 4.3 权限敏感度 → OpenFGA 数据域
按"角色 × 数据域"双维：**高敏域**（成本/毛利/底价、横向报价/客户名单、供应商报价、薪资、客户 PII）默认仅经理/老板/财务可见，其余角色**脱敏或拒答**。这批高敏标记直接转成 OpenFGA 的资源类型与关系。

### 4.4 连接器策略：系统无关，接口不规范就封装成 skill/MCP（本轮确认）
**我们不要求客户用某种 ERP**，目标是让 agent 接入企业**已有**的任何系统。按接口成熟度分三档接入，本质是"FSE 把每家企业现有系统封装成标准 MCP 工具"：

| 档 | 系统接口现状 | 接入方式 | 风险/说明 |
|---|---|---|---|
| A. 有规范 API | 金蝶云星空 WebAPI、用友云 OpenAPI、电商奇门等 | 直接封装成 **MCP Server**（REST→工具）| 最省力、最安全；优先 |
| B. 无 API / 老本地版 | 用友 U8/T3/KIS、管家婆、速达等 | **把数据库操作封装成 skill/MCP**（你的原话）：只读查询封装成参数化只读工具 | ⚠️ 见下方安全红线 |
| C. 无 API 且不宜直连库 | 纯 C/S 老软件、无库权限 | **桌面端 UI 自动化（FlaUI/RPA）兜底** | 最脆、最后手段；人工监督 |

**这正是 FSE 的核心工作**：上门把 B/C 档系统封装成标准 MCP 工具，让上层通用 agent 与场景模板不必关心底层是什么系统——**"系统无关"就是这样实现的**。

**封装数据库操作的安全红线（重要）**：
- **默认只读**：DB 封装的工具优先只做 `SELECT`；参数化查询，杜绝 SQL 注入；连接用**最小权限只读账号**。
- **写操作要极其谨慎**：直接写库会**绕过 ERP 的校验与业务逻辑**（触发器、库存扣减规则、审批流），极易搞坏数据。**能走官方 API 写就绝不直连库写**；必须直连库写时，强制人在环 + 审批 + 事务 + 可回滚 + 审计，且需与客户明确风险。
- **优先级**：A 档 API > B 档只读库 > C 档 UI 自动化；**写操作一律优先官方 API**。
- 每个封装工具都在 OpenFGA 里注册为受角色管控的资源（谁能调、能碰哪些数据）。

## 5. 开源方案再评估（决策 4：优先看开源）

加上"企微多租户 SaaS + 服务商生态 + 桌面 app"这几个约束后，候选进一步收窄。结论先行：**不存在能直接当产品基座的开源整机；核心必须自研，用开源做组件。**

### 5.1 为什么没有"直接拿来即用"的整机
我们最有差异化、最花钱的三层，**任何开源整机都没有现成的**：
1. **企微组织架构 → OpenFGA 权限同步**（免全量导入、复用既有角色/职位）——我们的核心竞争力，必自建 + 独立审计。
2. **多租户接入与隔离**（第三方应用授权链 + 按 corpid 隔离）+ **把各家异构系统封装成标准 MCP/skill**（含无 API 系统的 DB/UI 兜底）——必自建。
3. **按角色的 skill 体系 + 权限过滤检索**——只有 Open WebUI 原生有权限过滤检索，但它白标有授权限制。

选任何整机（Dify/FastGPT/LibreChat…）都得把上面三层重写，"基座"的意义就没了，反而背上它的架构包袱与许可证风险。

### 5.2 许可证红线（复述，务必守住）
- **❌ Dify / FastGPT**：改过的 Apache，**禁止未授权运营多租户** → 我们正是多租户 SaaS，直接违约。
- **❌ n8n**：仅内部用，不能卖衍生品。
- **⚠️ Open WebUI**：权限过滤 RAG 最好，但白标 >50 用户需企业授权。
- **✅ 可商用（MIT/Apache）**：LangGraph、LlamaIndex、OpenFGA、Qdrant、Tauri、Goose、AnythingLLM、LibreChat、RAGFlow、Docling。

### 5.3 推荐：核心自研 + 开源组件（都是 MIT/Apache 可商用）
| 层 | 选型 | 角色 |
|---|---|---|
| 编排 | **LangGraph**（MIT）| 确定性工作流骨架 + LLM 节点 |
| RAG 框架 | **LlamaIndex**（MIT）| 元数据过滤检索 |
| 权限 | **OpenFGA**（Apache）| ReBAC 核心 |
| 向量库 | **Qdrant**（Apache）| 权限预过滤检索 |
| 桌面壳 | **Tauri v2**（MIT/Apache）| 路径沙箱 |
| 文档解析 | **Docling / MarkItDown / RAGFlow**（可商用）| 文档→Markdown |
| 抽取 | **LangExtract**（Apache）| 可溯源结构化抽取 |

### 5.4 值得"读源码/借鉴"的整机（不直接用）
- **LibreChat（MIT）**：抄它的**每实体 ACL 权限模型**思路 + MCP 按用户上下文注入。
- **AnythingLLM（MIT）/ Goose（Apache）**：抄它们的**桌面 + 本地文件 + 本地 MCP**实现参考。
- **Open WebUI**：读它的**权限过滤检索**实现（注意授权，不直接用于产品）。

## 6. RAG 流水线：加入"文档→Markdown 解析层 + LangExtract 抽取层"

回答你的问题——你说的"Google md 文档方法"归位到流水线的**两个不同环节**，都建议采纳：

```
企业文档(PDF/Word/Excel/扫描件)
   │
   ├─[解析层] 文档 → Markdown  ← 你说的"md 方法"在这里
   │    · 首选大模型视觉转换(Gemini/Claude)：读扫描件、复杂版式、表格不散架
   │    · 开源兜底：Docling / MarkItDown / RAGFlow(批量、离线、私有)
   │    · 产出：保留标题/表格/列表结构的 Markdown
   │
   ├─[分块层] 语义分块 + 上下文检索(Anthropic contextual retrieval)
   │    · 每块预置 50-100 token 定位上下文；附 ACL/scope/tenant 元数据
   │
   ├─[抽取层] LangExtract(可选、按需)  ← Google 的这个库在这里
   │    · 从文档抽可溯源的结构化字段(交期/数量/单价/料号…)
   │    · 每字段绑定源文本字符位置 → 可高亮验证(契合"诚实可核验")
   │    · 产出：结构化事实 → 入库/喂工具，而非只做向量检索
   │
   ├─[存储] Qdrant 单集合 + payload 分区(tenant_id + scope + ACL)
   │
   └─[检索] 混合(BM25+向量) + rerank + 服务端 ACL 预过滤(filter-before-grounding)
```

**为什么解析层用 Markdown 很关键**：PDF/Excel 直接切块会把表格、标题层级切散，检索质量差；先转成保留结构的 Markdown 再切，表格、条款、层级都完整，检索命中率明显提升。制造/贸易大量是**带表格的单据（BOM、报价单、对账单、工艺卡）**，这一层收益尤其大。

**LangExtract 用在哪**：当 agent 需要的不是"找到相关段落"而是"抽出确定字段"时（如把 100 份 PDF 订单的交期/数量结构化）。它的**字符级溯源**正好支撑我们"诚实、可核验"的产品原则（对抗竞品"指标注水"的信任危机）。注意它标注"非 Google 官方支持产品"，Apache 2.0，可自托管（Ollama/Docker/Vertex）。

**两者分工一句话**：**Markdown 解析层**提升"检索"质量（入口）；**LangExtract 抽取层**提供"可溯源结构化事实"（按需）。不是二选一。
