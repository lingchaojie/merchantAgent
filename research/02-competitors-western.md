# Western / Global Enterprise AI Agent Platforms — Competitor & Architecture Research

**Research date:** 2026-07-08
**Purpose:** Architectural and permission-model reference for an enterprise agent platform that connects to a company's internal systems and helps employees across roles (sales, production, manager) boost productivity, with strict permission isolation.
**Focus dimension (most important):** how each platform ensures an employee only ever sees data they are authorized for — ACL-aware retrieval, identity propagation, and oversharing prevention.

> Method note: findings were gathered via extensive WebSearch/WebFetch against primary vendor documentation (learn.microsoft.com, help/developer/architect.salesforce.com, docs.glean.com, docs.aws.amazon.com, cloud.google.com, docs/servicenow.com) plus independent security analyses for vulnerability and honest-limits claims. Source URLs are inline. Uncertainty is flagged throughout. Product naming in this space churned heavily in 2025–2026 (see each section).

---

## The one pattern that matters (read first)

Every serious platform converges on the same core idea for "an employee only sees what they're allowed to see":

1. **Crawl/ingest source-system ACLs alongside content** (or call the source live), and
2. **Authenticate every query as a specific end-user identity**, then
3. **Intersect the user's resolved identity + group memberships against per-document ACLs at query time** — filtering *before* retrieved content reaches the LLM, so the model can never cite, summarize, or leak content the user lacks access to.

The universal weakness is equally consistent: **the platform only enforces the *existing* source permissions.** If the source is overshared ("Everyone"/"anyone with the link"/stale groups), the agent becomes a *force multiplier* that surfaces that latent exposure instantly. Hence every vendor now sells a **governance/DSPM layer** (oversharing detection, sensitivity labels, access reviews) on top of ACL-aware retrieval. A secondary, near-universal caveat is **ACL freshness**: most platforms enforce a *mirrored snapshot* of permissions refreshed on a crawl/sync schedule (minutes to 24h), not a live per-result check against the source.

Ranked by depth/maturity of the permission-aware retrieval model (mid-2026): **Glean ≈ Microsoft ≈ Amazon Q ≈ Google (all strong, ACL-at-query-time) > ServiceNow (strong but single-platform) > Salesforce (weakest — enforcement is largely delegated to the implementer).**

## 1. Glean — the reference design for permission-aware retrieval

**What it is.** "Work AI" platform combining permission-aware enterprise search, an AI assistant, and an agent-building/orchestration layer over a company's SaaS apps, grounded in a unified index + knowledge graph. This is the most architecturally relevant reference for the permission problem.

**How agents are built (low-code + pro-code).** A **unified Agent Builder** (Nov 2025) blends a visual drag-and-drop canvas (logic, conditions, error handling, templates, version history, rollback), a conversational natural-language builder, and a pro-code path (APIs/SDKs, hosted MCP servers, integrations with OpenAI Agents SDK, LangGraph, Bedrock Agents, Google ADK). Agents run on an LLM-agnostic engine with agentic looping and a Model Hub to mix models per step. ([agents blog](https://www.glean.com/blog/agents-product-blog), [AI Agent Builder](https://www.glean.com/ai-agent-builder))

**Connector model.** 100+ native connectors (marketing cites 275+); each connector does **data fetching** (content *plus* the source's permission map) and **content parsing**, and must expose three categories: **Content**, **People** (identities, roles, permissions, groups), and **Activity**. Three access modes: **Indexed** (default — content + permission snapshot mirrored into the tenant, read at query time), **Live** (call source during the request), and **Hybrid**. ([About connectors](https://docs.glean.com/connectors/about))

### Permission & security model (the core)
- **Crawl + store ACLs with content.** Connectors fetch the "permissions map from each source, ensuring search results strictly adhere to the access permissions set in the source application." ACLs are stored *alongside* content per-document in the customer's isolated tenant; each document carries a `visibility` field plus owner/assignedTo/updatedBy identities that "travel with each document, enabling access filtering." This is a **zero-copy / mirrored-permission** design — data stays in its original governance boundary. ([Data Flow](https://docs.glean.com/security/architecture/data-flow), [compliance vs ChatGPT Enterprise](https://www.glean.com/perspectives/understanding-gleans-compliance-features-compared-to-chatgpt-enterprise))
- **Unified identity schema.** Glean does not store each source's raw permission format; it maps all users into its **own identity schema of groups and memberships**. An **identity crawl** pulls users, groups, and members across sources; some groups are source-derived, others Glean-internal. Identity is anchored to the IdP via **directory sync/SCIM** — Glean *reads* group membership but "does not sync, modify, or enforce the IdP's own permission rules" (IdP stays authoritative). Result: source ACLs (in group/user terms) + a unified membership graph → a resolved set of groups per Glean user that filtering keys off. ([identity schema](https://www.glean.com/blog/using-our-identity-schema-to-deliver-personalized-permissions-aware-results), [group-based permissions](https://docs.glean.com/administration/identity/roles/group-based-permissions))
- **Concrete ACL structure (Indexing API):** per-document `permissions` object with `allowAnonymousAccess`, `allowAllDatasourceUsersAccess`, `allowedUsers` (by email or datasourceUserId), and `allowedGroups` (with **nested group** support). Users must be indexed before they can be granted; permissions process asynchronously; `/checkdocumentaccess` validates. This reveals the underlying model native connectors populate automatically: **allow-lists of users + (nested) groups per document.** ([Indexing API permissions](https://developers.glean.com/api-info/indexing/documents/permissions))
- **Query-time enforcement.** No anonymous search; SSO auth required. Flow: domain→tenant Query Endpoint (dedicated static IP per tenant) → SSO authenticate → resolve identity to groups/memberships → **filter results against per-document ACLs in the index** ("checks permissions per query"). Source app *also* re-enforces on click-through as defense-in-depth. For **indexed** connectors, enforcement uses the mirrored ACL snapshot (not a live source round-trip per result); live re-validation only for live/hybrid mode + click-through. ([Data Flow](https://docs.glean.com/security/architecture/data-flow), [Confluence connector](https://help.glean.com/connectors/native/confluence))
- **ACL freshness.** Marketing says "real-time"; docs qualify: SharePoint permission-only updates process in **<2 hours** with webhooks, up to **24 hours** under `Sites.Selected`. Identity crawls run periodically. Honest read: change *ingestion* is bounded by webhook/crawl latency; source click-through backstops staleness. ([SharePoint controls](https://docs.glean.com/connectors/native/sharepoint/security/controls))
- **Least-privilege crawling** (e.g., SharePoint `Sites.FullControl.All` is required only to *read* ACL endpoints; `User.Read.All`+`GroupMember.Read.All` to enforce permissions). Agents/Assistant inherit the same envelope: "only see data and take actions the user already has access to," cite only permitted sources. ([Protect overview](https://docs.glean.com/administration/protect/overview))
- **Governance/DSPM — Glean Protect / Protect+:** point-in-time and (Protect+) weekly/continuous oversharing scans across 100+ sources, auto-hide of sensitive content, AI guardrails (prompt-injection, jailbreak, toxic-content, agent-alignment checks), sensitive-content classifiers. Findings are themselves access-restricted. This is Glean's answer to the "can-access ≠ need-to-know" gap. ([Protect](https://docs.glean.com/administration/protect/overview), [governance](https://www.glean.com/product/governance))
- **Isolation/crypto:** single-tenant per customer (dedicated GCP project + static IP); AES-256 at rest, TLS 1.2+; key rotation 30d (GCP)/90d (AWS); zero LLM data retention; SOC 2 Type II, ISO 27001, ISO 42001. ([about deployment](https://docs.glean.com/get-started/prepare/about-deployment), [legal](https://www.glean.com/legal))

**RAG design.** Unified index + **Knowledge Graph** (Content/People/Activity pillars, real-time model) with **hybrid search** (dense vectors + lexical), permission-filtered before generation, cited answers. ([Knowledge Graph](https://docs.glean.com/security/knowledge-graph), [hybrid vs vector](https://www.glean.com/blog/hybrid-vs-rag-vector))

**Multi-role behavior.** Personalization via People/Activity + a Personal Graph; role differences emerge from the graph + ACLs rather than hand-written rules ("an engineer sees only the documents their role permits… because the graph inherited and enforces those boundaries"). Collective insights gated by privacy thresholds. Agent creation governed via library + versioning. ([enterprise context](https://www.glean.com/enterprise-context))

**Deployment.** Glean-hosted SaaS (GCP) or **Customer-Hosted** (managed service inside customer's own GCP/AWS VPC for data-residency; not classic self-host — Glean retains minimal operate access). No Azure host option; Azure OpenAI available as a *model* provider. ([about deployment](https://docs.glean.com/get-started/prepare/about-deployment))

**Strengths.** Best-in-class ACL-aware retrieval; uniform enforcement across Search/Assistant/Agents; strong isolation/compliance; governance built into the retrieval layer; knowledge-graph grounding.
**Weaknesses.** (1) **Can-access ≠ need-to-know** — faithfully enforcing overly-permissive source ACLs just delivers overshared content faster (base guarantee; oversharing scanning is the paid Protect+ add-on). (2) **Aggregation risk** — RAG can synthesize individually-permitted fragments into sensitive aggregate context. (3) **Stale-ACL window** for indexed connectors despite "real-time" marketing. (4) Dependent on connector fidelity + broad read scopes. (5) No Azure host; Customer-Hosted forbids customer patching. (Independent critique: [Knostic](https://www.knostic.ai/blog/glean-data-security).)

## 2. Microsoft Copilot Studio + Microsoft 365 Copilot

**Defining principle.** Copilot introduces **no new data access** — it inherits and *accelerates* the tenant's existing permission model. Nearly every strength and weakness flows from this. ([architecture](https://learn.microsoft.com/copilot/microsoft-365/microsoft-365-copilot-architecture))

**How agents are built.** Two approaches: **declarative agents** (instructions + knowledge + actions, running on Copilot's orchestrator/models; built via Agent Builder, Copilot Studio, M365 Agents Toolkit, or SharePoint) and **custom engine agents** (bring your own orchestrator/models; M365 Agents SDK or Azure/Microsoft Foundry). Low-code building blocks in Copilot Studio: **topics, tools/actions, knowledge sources, triggers, generative orchestration** (LLM planning layer, now default), plus **autonomous** and **multi-agent** (child/connected agents, A2A). Pro-code: **M365 Agents SDK** (successor to the sunset Bot Framework SDK), **Foundry Agent Service**, **Agents Toolkit**, **SharePoint agents**. **Agent Store** offers Microsoft/partner/own agents (e.g., Researcher, Analyst — GA Jun 2025). ([agents overview](https://learn.microsoft.com/en-us/microsoft-365/copilot/extensibility/agents-overview), [Copilot Studio](https://learn.microsoft.com/en-us/microsoft-copilot-studio/fundamentals-what-is-copilot-studio))

**Connector model — INDEXED vs LIVE (key distinction).** **Copilot connectors** (formerly Graph connectors) crawl external data (ServiceNow, Salesforce, Confluence, file shares, DBs) *into* Microsoft Graph + the semantic index (with citations); 100+ prebuilt + a Connectors API; **federated connectors** (preview) query in real time. **Power Platform connectors** call line-of-business systems *live* (actions/triggers, no replication). **MCP is GA in Copilot Studio (May 2025).** ([indexed vs live](https://learn.microsoft.com/en-us/microsoft-copilot-studio/knowledge-graph-vs-power-platform-connectors), [MCP GA](https://www.microsoft.com/en-us/microsoft-copilot/blog/copilot-studio/model-context-protocol-mcp-is-now-generally-available-in-microsoft-copilot-studio/))

### Permission & security model (the core)
- **Query-time security-context enforcement.** Verbatim: "**All Microsoft 365 Copilot prompts run in the security context of the user who initiates the prompt.** … for a user to see an item in a prompt response, they must first have the necessary permissions to access the content." RAG flow: prompt → orchestrator → grounding against Graph + semantic index **permission-trimmed to the user** → LLM → cited response. Honors **Conditional Access + MFA**. Prompts/Graph data **not used to train foundation LLMs.** ([DLP for Copilot](https://learn.microsoft.com/en-us/purview/dlp-microsoft365-copilot-location-learn-about), [architecture](https://learn.microsoft.com/copilot/microsoft-365/microsoft-365-copilot-architecture))
- **The OVERSHARING problem (defining risk).** Because Copilot enforces *existing* not *intended* permissions, Microsoft's own FastTrack team calls it a **"force multiplier"** for latent access (anyone-with-the-link, stale groups, over-permissioned sites, missing labels). This is *the* core governance issue. ([FastTrack risk mapping](https://techcommunity.microsoft.com/blog/fasttrackblog/understanding-copilot-risk-mapping-exposure-across-zero-trust-pillars/4534183), [mitigate oversharing](https://techcommunity.microsoft.com/blog/microsoft365copilotblog/mitigate-oversharing-to-govern-microsoft-365-copilot-and-agents/4448744))
- **Oversharing control stack** (three pillars: remediate oversharing / set up guardrails / meet regulations):
  - **Restricted SharePoint Search (RSS)** — admin allow-list (~100 sites), explicitly a short-term stopgap; **new enablement blocked from Jul 31, 2026** → migrate to Restricted Content Discovery.
  - **Restricted Content Discovery (RCD)** — the successor; suppresses site content from org-wide search + Copilot (unless recently interacted), without removing from the index or changing direct access. ([RCD](https://learn.microsoft.com/en-us/sharepoint/restricted-content-discovery))
  - **SharePoint Advanced Management (SAM)** — now bundled with the Copilot license; Data Access Governance reports, automated **site access reviews**, Restricted Access Control.
  - **Microsoft Purview:** **sensitivity labels + encryption** honored at grounding time ("the user must have EXTRACT and VIEW usage rights for Copilot to interact with the content"); labels flow to *outputs* (response reflects highest-priority label; generated content inherits it). **DLP for Copilot** — label-keyed policy means "content … isn't processed by Copilot or used in the response summary." **DSPM for AI** — discovery + **oversharing/data-risk assessments** across Copilot, agents, Foundry, and third-party AI (ChatGPT/Claude/Gemini). Plus eDiscovery, retention, Insider Risk. ([data protection](https://learn.microsoft.com/en-us/microsoft-365-copilot/microsoft-365-copilot-architecture-data-protection-auditing), [Purview for AI](https://learn.microsoft.com/en-us/purview/ai-microsoft-purview))
  - **Copilot Control System (CCS)** — umbrella governance framework. ([CCS](https://learn.microsoft.com/en-us/microsoft-365/copilot/copilot-control-system/security-governance))
- **Identity propagation & agent identity.** M365 Copilot/declarative agents act **as the signed-in user** (delegated Entra identity). Copilot Studio agents can use Entra/OAuth2/**No authentication** (the last creates an unauthenticated endpoint — a documented risk) and can run tools with the user's credentials. **Microsoft Entra Agent ID** auto-assigns every Studio/Foundry agent an Entra identity (Conditional Access + lifecycle for non-human identities); **Microsoft Agent 365** (Ignite Nov 2025) is the emerging control plane for agent sprawl. ([end-user auth](https://learn.microsoft.com/en-us/microsoft-copilot-studio/configuration-end-user-authentication), [Entra Agent ID](https://learn.microsoft.com/en-gb/entra/agent-id/what-is-microsoft-entra-agent-id))
- **ACLs on indexed external data.** Graph-connector items carry a per-item **ACL** (array of Entra user/group entries; an `Everyone` principal exists; **deny > grant**). ACLs are set **at ingestion time**, so a misconfigured connector (`Everyone`) is a direct oversharing vector. ([manage items](https://learn.microsoft.com/en-us/graph/connecting-external-content-manage-items))
- **Isolation/residency.** Data stays in the M365 service boundary + tenant geography; GDPR + EU Data Boundary — **but Anthropic models (now used in some Copilot experiences) are out of scope for the EU Data Boundary** (a real compliance nuance). Copilot Studio adds CMK + Customer Lockbox. ([privacy](https://learn.microsoft.com/en-us/microsoft-365/copilot/microsoft-365-copilot-privacy))

**RAG design.** Lexical + **semantic index** built from Graph (permission-aware, "all organizational boundaries within your tenant are respected"); a **Retrieval API** exposes the same permission-honoring hybrid index. Limitation: short snippets weaken whole-document/multi-table reasoning. ([semantic index](https://learn.microsoft.com/en-us/microsoftsearch/semantic-index-for-copilot))

**Multi-role behavior.** Scoped department agents (HR-site SharePoint agent, Employee Self-Service) naturally differ per-user via knowledge-source scoping + per-user enforcement. **Who can build/publish** governed via Power Platform: zoned governance, Managed Environments, publishing approval by an AI Admin, connector DLP (note: "by default every connector is available to every maker"). ([sec-gov](https://learn.microsoft.com/microsoft-copilot-studio/guidance/sec-gov-phase2))

**Deployment.** SaaS in M365. M365 Copilot ~$30/user/mo (annual); **Copilot Chat** metered pay-as-you-go for agents; Copilot Studio via credit packs/PAYGO. ([billing](https://learn.microsoft.com/en-us/microsoft-copilot-studio/billing-licensing))

**Strengths.** Permissions inherited not reinvented; deepest native governance surface (labels, DLP-for-Copilot, DSPM, Conditional Access, Lockbox, Entra Agent ID); data-boundary + no-training guarantees; labels flow to outputs.
**Weaknesses.** (1) **Oversharing is the defining weakness** — latent permission debt becomes instantly exploitable; the entire RSS/RCD/SAM/DSPM toolset exists because tenants are widely overshared. (2) Heavy admin burden + E5/SAM licensing for best controls, amplified by rapid rename/retire churn (RSS retiring Jul 2026, Bot Framework sunset, Foundry/Agent 365 renames). (3) **Prompt-injection/exfiltration surface** — **EchoLeak (CVE-2025-32711, CVSS 9.3)** was a zero-click indirect-prompt-injection data-exfil flaw (patched server-side ~May 2025); the *class* of risk is structural. (4) EU-Data-Boundary gap for Anthropic models. (5) "No authentication" agents + default-open connectors shift safety onto admin discipline. ([EchoLeak](https://www.bleepingcomputer.com/news/security/zero-click-ai-data-leak-flaw-uncovered-in-microsoft-365-copilot/))

## 3. Amazon Q Business

> **Critical status (mid-2026):** Q Business **closes to new customers on Jul 31, 2026** (existing customers keep bug/security fixes, no new features). AWS steers new adopters to **Amazon Quick (Quick Suite)** via a "Bring Your Own Index" migration — and **Q Business guardrails do NOT carry over through BYOI**. Treat Q Business as "stable but sunset." ([availability notice](https://docs.aws.amazon.com/amazonq/latest/qbusiness-ug/qbusiness-availability-change.html))

**What it is.** Fully managed generative-AI assistant with **permission-aware responses + citations** — a packaged RAG-plus-actions product, not a build-your-own-agent framework. Built from: Application → Index → Retriever → Data sources (connectors) → Web experience → Plugins. ([prescriptive guidance](https://docs.aws.amazon.com/prescriptive-guidance/latest/retrieval-augmented-generation-options/rag-fully-managed-q-business.html))

**How agents/apps are created.** **No-code: Amazon Q Apps** (turn a chat into a reusable app, or describe it in natural language; Pro tier). **Pro-code:** `Chat`/`ChatSync`/`SearchRelevantContent` APIs; ISVs can become a **data accessor** to query a customer's index on the user's behalf. **Plugins/actions:** built-in (Jira, ServiceNow, Salesforce, PagerDuty) + **custom via OpenAPI**; runtime auto-orchestration (Pro tier). Runs on Bedrock models but **you cannot pick the model** (opaque, unlike building on Bedrock directly). ([Q Apps](https://aws.amazon.com/about-aws/whats-new/2024/07/amazon-q-apps), [actions](https://docs.aws.amazon.com/amazonq/latest/qbusiness-ug/actions.html))

**Connector model.** ~27–30 documented connectors (SharePoint editions, Salesforce, ServiceNow, Confluence, S3, Slack, DBs, etc. — "40+" is soft marketing). Retriever = native Q index or **Amazon Kendra**. Connectors crawl documents **and ACL/identity metadata**; **Custom Document Enrichment** transforms content; sync is scheduled and must re-run to capture ACL changes. ([connectors list](https://docs.aws.amazon.com/amazonq/latest/qbusiness-ug/connectors-list.html))

### Permission & security model (the core) — three layers
- **(A) Identity/auth.** **AWS IAM Identity Center** (recommended) is the authoritative directory + trusted-token-issuer broker; external IdPs (Okta/Entra/Ping) federate via SAML/OIDC, users/groups provisioned via **SCIM 2.0**. Alternative: **IAM Federation** direct to IdP (weaker propagation; **user-level topic controls unsupported under SAML/OIDC federation**). ([API calls](https://docs.aws.amazon.com/amazonq/latest/qbusiness-ug/making-sigv4-authenticated-api-calls.html))
- **(B) Trusted identity propagation (the token-exchange mechanics)** — how retrieval is filtered per *real end user*: user signs in at IdP → app gets OIDC JWT (`iss` must match a **trusted token issuer** in Identity Center, `aud` must match) → `CreateTokenWithIAM` exchanges it for an Identity Center token carrying **`sts:identity_context`** → STS `AssumeRole` with `ProvidedContexts` yields **identity-aware SigV4 credentials** → `ChatSync`/`SearchRelevantContent` now know the specific user + groups and log identity to CloudTrail. The hosted web experience does this automatically. **Critical caveat: only group memberships present in Identity Center are passed — NOT external-IdP groups — so you must SCIM-sync all relevant users AND groups into Identity Center or group-based ACL filtering silently under-matches.** ([trusted identity propagation](https://aws.amazon.com/blogs/machine-learning/configuring-amazon-q-business-with-aws-iam-identity-center-trusted-identity-propagation/))
- **(C) ACL-aware RAG.** At ingestion, connectors index the document *and* its principals — **user email, local group name, federated group name** — into the **User Store** (users↔groups↔access), populated by the **Identity Crawler**. At query time, Q resolves the authenticated user → User Store identity + groups, and **filters candidate documents to only those whose ACL principals match** before generating. Two users, same question, different answers. Hard rules: **ACL crawling is ON by default**; docs indexed *without* ACLs are treated as **fully public**; once enabled, ACL crawling **can't be turned off**; emails case-insensitive (Dec 2024); **ACLs only as fresh as the last sync**. Source ACLs must be **principal-mapped** to Identity Center identities (User Store APIs are a flagged **privileged operation** — mistakes cause "unacceptable changes in accessibility"). ([connector concepts/authorization](https://docs.aws.amazon.com/amazonq/latest/qbusiness-ug/connector-concepts.html), [User Store](https://docs.aws.amazon.com/amazonq/latest/qbusiness-ug/principal-store-hiw.html))
- **Guardrails/admin controls.** Global: allow/deny uploads, toggle personalization, **restrict responses to enterprise data only vs. LLM world-knowledge fallback**, hallucination checking. **Topic-level controls** (max 2 topics, ≤5 rules each): block or answer-from-enterprise-data, target by user/group via inclusion *or* exclusion, **most-restrictive-wins**. ([guardrails](https://docs.aws.amazon.com/amazonq/latest/qbusiness-ug/guardrails.html))
- **Data protection.** At rest: AWS-owned KMS key or **customer-managed CMK**; in transit TLS 1.2+ (FIPS endpoints). **No training on customer data.** Per-user identity logged to CloudTrail. ([data protection](https://docs.aws.amazon.com/amazonq/latest/qbusiness-ug/data-protection.html), [service improvement](https://docs.aws.amazon.com/amazonq/latest/qbusiness-ug/service-improvement.html))

**RAG design.** Native Q index or Kendra; capacity metered in index units (20k docs / 200MB each); citations returned; response length by tier; admin can force enterprise-only grounding. ([select retriever](https://docs.aws.amazon.com/amazonq/latest/qbusiness-ug/select-retriever.html))

**Multi-role.** Per-user filtering is the default posture (roles emerge from source ACLs); topic controls scope per user/group; admins manage the Q Apps library. ([tiers](https://docs.aws.amazon.com/amazonq/latest/qbusiness-ug/tiers.html))

**Deployment.** Managed AWS service; US/EU regions (some Pro features limited in Ireland/Sydney); VPC support; **Lite $3/user/mo**, **Pro $20/user/mo**, index ~$0.14–0.264/unit-hr. ([pricing](https://aws.amazon.com/q/business/pricing/))

**Strengths.** Best-in-class *managed* permission-aware RAG (ACL-crawl + User Store + trusted identity propagation is more turnkey than hand-rolling on Bedrock); ACL crawling on by default; strong governance defaults (CMK, no-training, CloudTrail per-user audit, enterprise-only grounding toggle); low-code + pro-code surfaces.
**Weaknesses.** (1) **Sunset service** (closed to new customers Jul 31, 2026). (2) **ACL/identity-mapping burden is real and unforgiving** — correctness depends on enabling ACL crawl (irreversible), SCIM-syncing groups into Identity Center (external-IdP groups NOT auto-passed), correct principal mapping, and regular re-syncs; **misconfig can silently expose restricted docs with no alert** ([BeyondScale](https://beyondscale.tech/blog/amazon-q-business-enterprise-security-guide)). (3) Federation feature gaps (no user-level topic rules under SAML/OIDC). (4) Only 2 topic controls; guardrails don't migrate to Quick. (5) No model choice; connector count oversold; some connectors preview-only. *(Note: the 2026 Wiz CVEs affected Amazon Q **Developer**, not Q Business.)*

## 4. Google Agentspace / Gemini Enterprise

> **Naming/lineage (essential):** **Agentspace** (Dec 2024) → folded into **Gemini Enterprise** (Oct 2025) → at Cloud Next '26 (Apr 2026) Vertex AI rebranded **"Gemini Enterprise Agent Platform" (GEAP)** and Vertex AI Search → **"Agent Search."** "Agentspace" is effectively deprecated as a brand; the deepest ACL docs still live under legacy `generative-ai-app-builder`/`discoveryengine` paths. ([introducing Gemini Enterprise](https://cloud.google.com/blog/products/ai-machine-learning/introducing-gemini-enterprise), [the new Gemini Enterprise](https://cloud.google.com/blog/products/ai-machine-learning/the-new-gemini-enterprise-one-platform-for-agent-development))

**How agents are built.** **No-code Agent Designer** (natural language, schedule/trigger-based) → **low-code Agent Studio** → **pro-code Agent Development Kit (ADK)** (graph-based sub-agent networks) + **Agent Engine/Runtime**. Prebuilt Google agents: Deep Research, NotebookLM Enterprise, Idea Generation, Data Science. Interop via **A2A, MCP, AP2**. ([Agentspace launch](https://cloud.google.com/blog/products/ai-machine-learning/google-agentspace-enables-the-agent-driven-enterprise))

**Connector model.** Backbone = **Vertex AI Search / Agent Search (Discovery Engine)**. Two access models: **Ingestion** (copied into index; better quality + Knowledge Graph) vs **Federation** (live, not copied; lower quality). Sync pulls **identity data (roles, permissions, users) + entity data**; **identity full syncs as often as every 30 min**, but **incremental syncs skip identity + deletions** → ACL changes propagate on the identity-sync schedule, not instantly. Connectors: Drive/Gmail/BigQuery (1st-party); Entra ID, OneDrive, SharePoint, Jira, Confluence, ServiceNow, Salesforce (Sales Cloud only), Slack, Box, Workday (3rd-party). ([connectors/data stores](https://docs.cloud.google.com/agentspace/agentspace-enterprise/docs/introduction-to-connectors-and-data-stores))

### Permission & security model (the core) — two layers + query-time filtering
- **Two distinct layers (don't conflate):** (1) **IAM access control** governs admin/API access to the search *resources* (`discoveryengine.admin/editor/user/viewer`; end users get a **custom role** with just serving permissions — `servingConfigs.search`/`.answer` — granted to one Google group). (2) **Data source access control (document ACLs)** governs *which indexed docs a user may see*. IAM decides *whether you can call search*; the ACL layer decides *what you get back*. ([access control](https://docs.cloud.google.com/generative-ai-app-builder/docs/access-control), [data-source access control](https://docs.cloud.google.com/generative-ai-app-builder/docs/data-source-access-control))
- **Identity establishment.** Three paths: Google Identity (native or third-party-IdP synced to Cloud Identity), or **third-party IdP federation without sync — requires Workforce Identity Federation**. **Only ONE identity provider per Agent Search location** (a real multi-IdP constraint).
- **Identity mapping.** For federation, **`google.subject` must map to email**; **`google.groups` carries group membership from the IdP assertion** (Entra/Okta OIDC: `assertion.email` + `assertion.groups`). Group IDs become principals for ACL matching — the IdP tells Google the groups; Google doesn't recompute them.
- **Document ACLs.** Per-document `acl_info` with `readers → principals` (`user_id`/`group_id`). Cloud Storage/BigQuery supply `acl_info` in metadata; **Drive + third-party sources pull ACLs automatically** ("no additional configuration"), retaining source ACLs so users are "only returned data to which they have access in the source system." **Access control must be enabled at data-store creation and cannot be toggled later.**
- **Query-time enforcement.** User authenticates (Google OAuth token, or short-lived Workforce Identity Federation token) → Google resolves `google.subject` (email) + `google.groups` into principals → **retrieval engine matches them against each candidate doc's `acl_info.readers.principals`; only docs whose readers include the user/group are returned** and reach grounding — the LLM never sees unauthorized content. Write **actions** run "under the permissions of the logged-in user for the target application." *(Google documents the principal-matching model but not the internal pre/post-vector filtering algorithm — inferred.)*
- **Hard limits/caveats.** **3,000 readers max per document**; one IdP per location; ACL enablement creation-time only; **staleness risk** (identity syncs ≤30 min but incrementals skip identity — revoked access can linger); **correctness depends entirely on source ACLs** (over-shared source docs leak straight through; onus on content owners).
- **Broader security.** No training on customer data without permission + **zero-data-retention option**; sensitive-data (PII/PHI) scanning to block assets from agents; CMEK; VPC Service Controls, Access Transparency, data residency (Standard/Plus editions); **FedRAMP High**. 2026 agent-layer governance: **Agent Identity** (crypto ID per agent), **Agent Gateway**, **Model Armor** (prompt-injection/data-leak protection). ([zero-data-retention](https://docs.cloud.google.com/gemini-enterprise-agent-platform/resources/zero-data-retention), [FedRAMP](https://cloud.google.com/blog/topics/public-sector/google-agentspace-achieves-fedramp-high-authorization))

**RAG design.** Vertex AI Search / Agent Search engine + a **per-customer Enterprise Knowledge Graph** (employees↔teams↔docs↔accessible systems); ingestion builds vector index, serving does query-embedding → vector retrieval → **permission-aware filtering** + safety filters → grounded cited answer; blended/multimodal + Grounding with Google Search. ([RAG reference architecture](https://cloud.google.com/architecture/rag-genai-agentspace-vertexai))

**Multi-role.** **Agent Gallery** catalog (admin-provisioned); scoped agents per team via Agent Designer; secure request-and-approval gateway; single governance control plane; edition-scoped (Frontline users can only *use* provisioned agents). ([editions](https://docs.cloud.google.com/gemini/enterprise/docs/editions))

**Deployment.** GCP editions — **Business** (1–500 seats, permission-aware search, no full connector set), **Standard** (full connectors, enterprise security/compliance), **Plus** (latest models), **Frontline** (deskless). Data residency global/US/EU. **On-prem/sovereign via Google Distributed Cloud (GDC air-gapped GA)** — a genuine differentiator for government/defense. VPC-SC/CMEK/FedRAMP live in Standard/Plus.

**Strengths.** Genuinely permission-aware retrieval with clean two-layer model + automatic ACL inheritance for Drive/third-party; standards-based IdP federation (Entra/Okta/Ping via `google.subject`/`google.groups`); strong sovereignty story (FedRAMP High, air-gapped GDC); no-training + zero-retention; full-stack + agent-layer governance (Agent Identity/Gateway/Model Armor).
**Weaknesses.** (1) **Rebrand churn** (Agentspace→Gemini Enterprise→GEAP in ~18 months; mixed docs; perpetual "preview"). (2) Security only as good as source ACLs. (3) **ACL freshness is schedule-bound** (incrementals skip identity → revocation lag). (4) Hard limits: 3,000 readers/doc, one IdP/location, no ACL retrofit. (5) Federation setup friction (tokens, Secret Manager, CORS); Salesforce Sales-Cloud-only. (6) Query-time filtering *algorithm* internals not published.

## 5. ServiceNow AI Agents / Now Assist

**Central design premise.** Agents run **inside the Now Platform's existing security fabric**, not beside it — they reuse the platform's decades-old contextual-security engine (ACLs, roles, user criteria, domain separation) and add two agent-specific controls (invocation ACL + execution identity). Strongest where all data already lives on ServiceNow; the model is single-platform by nature.

**How agents are built.** Layered stack: **Now Assist** (GenAI assistant across ITSM/CSM/HRSD/Creator) → **AI Agents + AI Agent Studio** (build/configure; thousands of prebuilt agents; GA to all Now Assist customers in the **Yokohama release, Mar 2025**) → **AI Agent Orchestrator** (multi-agent) → **AI Agent Control Tower** (governance) → **Now Assist Skill Kit (NASK)** (custom generative skills). Low-code via AI Agent Studio/Flow Designer; pro-code via the **Fluent SDK** (`AiAgent`, `AiAgenticWorkflow`, tools, triggers, ACL deployment). Role-scoped "**AI Specialists**" (IT vs HR). ([Yokohama AI Agents](https://www.servicenow.com/company/media/press-room/yokohama-release-ai-agents.html), [Building AI Agents Guide](https://servicenow.github.io/sdk/guides/building-ai-agents-guide))

**Connector model.** **Workflow Data Fabric** (real-time unified data access) + **AI Agent Fabric** (cross-vendor via **MCP + A2A + A2UI**); a GA **MCP Server ("Action Fabric")** exposes governed actions to any AI agent; deep Microsoft + Google Cloud connectors; **AI Gateway** governs all MCP transactions. (2025–26 messaging shifted from legacy IntegrationHub spokes to MCP/A2A.) ([MCP protocol](https://www.servicenow.com/community/ceg-ai-coe-articles/mcp-the-protocol-powering-agentic-ai/ta-p/3508887), [AI Agent Fabric](https://www.servicenow.com/platform/ai-agent-fabric.html))

### Permission & security model (the core)
- **Foundation — platform ACL engine agents inherit.** **Deny-by-default/zero-trust**: no grant (or empty/invalid ACL) = denied. Each ACL rule pairs an object (table/field/record) with an operation (read/write/create/delete) and requires role(s) + condition(s) + optional script. These platform controls — **ACLs, encryption, user criteria, domain separation — automatically extend to all Now Assist apps** (e.g., if user criteria hide a KB article, Now Assist can't surface it). ([configuring ACLs](https://www.servicenow.com/community/platform-privacy-security-blog/configuring-acls-the-right-way/ba-p/3446017))
- **Two agent-specific controls (the heart):**
  - **(A) Invocation — `securityAcl` (who can call the agent):** mandatory on `AiAgent`/`AiAgenticWorkflow`; auto-generates `sys_security_acl` records; access = *Any authenticated user* / *Specific role* / *Public*. Maps to the AI Agent Studio UI making **role-based access a required, non-bypassable step**.
  - **(B) Execution identity — `runAsUser` / `dataAccess` (whose permissions apply):** set **either** `runAsUser` (fixed service identity, regardless of caller) **or** `dataAccess` (**runs as the invoking user**, restricted to listed roles). With `dataAccess`, **the ACL engine evaluates every data operation against the invoking user's effective permissions** — the agent literally cannot read/write a record the requester couldn't. Record-triggered agents can derive run-as identity from a column or a `runAsScript`.
  - **(C) Tool-level — `GlideRecordSecure`:** the guide mandates `GlideRecordSecure` (enforces ACLs) over plain `GlideRecord` (bypasses them), keeping ACL enforcement in the loop even in pro-code. ([Building AI Agents Guide](https://servicenow.github.io/sdk/guides/building-ai-agents-guide))

  > This **invocation-vs-execution split** is the key architectural fact: *who may trigger* an agent is governed independently from *whose data rights* its actions run under, and the default path binds execution to a real user identity, not an all-powerful bot account.
- **Retrieval-time ACL enforcement (AI Search RAG).** Now Assist in AI Search runs **Retrieval → Augmentation → Generation**; the Retrieval step "enforc[es] access controls … by evaluating user access to indexed tables and records." **Content security is automatically enabled and isn't configurable** — two users, same question, different source pools. **Known gap: AI Search does NOT support scripted field-level ACLs.** ([under the hood: AI Search](https://www.servicenow.com/community/now-assist-articles/under-the-hood-now-assist-in-ai-search/ta-p/2642915))
- **Domain separation (multi-tenant isolation).** Partitions one instance into logical domains; tenants can't query domains they lack access to. Supported in Now Assist Virtual Agent + Admin console — **but it's manual/high-overhead, and full support for the newest autonomous AI Agents is unconfirmed.** ([domain separation](https://docs.servicenow.com/bundle/rome-platform-administration/page/administer/company-and-domain-separation/concept/bp-what-is-domain-separation.html))
- **Now Assist Guardian (runtime guardrails):** sensitive-topic detection, offensiveness detection (SLM scans input+output, 16 categories), **prompt-injection detection** (direct + indirect via notes/comments). ([Guardian guide](https://www.servicenow.com/community/now-assist-articles/nowassist-guardian-a-comprehensive-guide-for-safer-ai-use-in/ta-p/3295827))
- **Data privacy.** **Data Privacy for Now Assist** does **real-time two-way PII masking** (placeholder to the LLM, restored after) — **but it's off by default**. Model training on customer data is **opt-in only**; Azure OpenAI (a hosting option) doesn't train on prompts. ([sensitive-data FAQs](https://www.servicenow.com/community/now-assist-articles/sensitive-data-and-advanced-ai-products-faqs/ta-p/3200663))
- **Governance — AI Control Tower** (GA ~Aug 2026): Discover/Observe/Govern/Secure/Measure; NIST + EU AI Act risk frameworks; **Veza** integration for least-privilege + a real-time **"kill switch"**; **Traceloop** runtime observability. ([AI Control Tower](https://newsroom.servicenow.com/press-releases/details/2026/ServiceNow-expands-AI-Control-Tower-to-discover-observe-govern-secure-and-measure-AI-deployed-across-any-system-in-the-enterprise/default.aspx))

**RAG design.** AI Search RAG (ACL-filtered retrieval → augmentation with top KB article → NowLLM → generation); retrieval pool scoped via "Approved for Now Assist" field + relevancy rules; **citations** link to source articles.

**Multi-role.** Role-scoped AI Specialists (HR agent bounded by HR roles/ACLs, ITSM by ITSM), enforced technically by `securityAcl` + `dataAccess` roles + domain separation.

**Deployment.** ServiceNow cloud (SaaS); during processing data goes to ServiceNow regional DCs or same-region Azure then is deleted. LLM: **NowLLM** (Apriel/NVIDIA Nemotron via NIM) or **BYO-LLM** (Azure OpenAI, Claude-via-AWS, Google). **No on-prem Now Assist** (unconfirmed/unavailable).

**Strengths.** Security model is the standout — reuses a mature contextual-security engine + binds agent execution to a real user identity (`dataAccess` + `GlideRecordSecure`); per-user retrieval-time ACL enforcement; data stays in one governed system of record (avoids the multi-source permission-reconciliation problem); unusually complete layered governance (Guardian + Control Tower + Veza kill switch + Traceloop); open interop (MCP/A2A/A2UI).
**Weaknesses.** (1) **Accuracy/autonomy limits** — summarization reliable, but autonomous cross-system resolution weaker (ServiceNow's own playbook concedes this). (2) **Configuration complexity** / shelfware risk. (3) Security caveats: **no scripted field-level ACLs in AI Search**, domain separation manual + unconfirmed for newest agents, **PII masking off by default**. (4) **Pricing controversy** (usage-based pivot, 20–30% uplift; stock fell ~40–50% H1 2026). (5) Cloud-only; no on-prem; heavy ServiceNow/Azure/NVIDIA concentration.

## 6. Salesforce Agentforce

> **Terminology drift:** Data Cloud→**Data 360** (Oct 2025); stack→**Agentforce 360** (Oct 2025); agent "Topics"→**"Subagents"** and Topic Selector→**Agent Router** (Apr 2026). Classic terms used below.

**Honest headline:** Agentforce's security depends heavily on **correct customer configuration** — the LLM enforces nothing, and several enforcement layers behave differently for agents than most assume. It is the **weakest** permission model of the six on a *default* basis.

**How agents are built.** The **Atlas Reasoning Engine** ("brain" — plans, evaluates, re-plans) drives agents defined by **Topics** (NL-classified job categories that scope the agent) + **Instructions** (NL rules) + **Actions** (Flows, Apex, Prompt Templates, MuleSoft/external services). Low-code **Agent Builder** (declarative, NL agent creation); pro-code **Agentforce DX** (VS Code/CLI, Testing API), **Python SDK**, and **Agent Script** (Oct 2025 — compiled declarative language for **deterministic** control + **hybrid reasoning**, a direct response to LLM unreliability). ([Atlas](https://www.salesforce.com/blog/atlas-reasoning-engine/), [hybrid reasoning](https://developer.salesforce.com/blogs/2025/10/introducing-hybrid-reasoning-with-agent-script))

**Connector model.** **Data Cloud/Data 360** is the data backbone (unifies structured+unstructured, powers RAG indexing/grounding). **MuleSoft** connects external systems + can expose them as **MCP servers** + an **AI Gateway**. **Native MCP** (Agentforce 3, Jun 2025) + **A2A** + **AgentExchange** marketplace. Grounding via **Retrievers over Search Indexes** (Hybrid Search recommended). ([MCP](https://developer.salesforce.com/blogs/2025/06/introducing-mcp-support-across-salesforce))

### Permission & security model (the core — most caveated of the six)
- **Two-layer identity model: the agent's running user ≠ the end user.** Every agent ties to a **running/"Agent User" identity** whose profile, permission sets, FLS, and sharing/record access set the **maximum scope**. **Agents do NOT run as the end user by default** — on public sites the agent runs as the **Guest User**; Apex actions run under a **platform integration user**. New agents start with **zero permissions**. ([secure implementation](https://www.salesforce.com/blog/best-practices-for-secure-agentforce-implementation-2/))
- **Identity-propagation gap.** Salesforce's *aspirational* engineering blog claims end-user identity "travels through all downstream interactions" via a "unified enforcement layer" — but architect + independent sources describe end-user identity propagation as a **problem you must design for** (e.g., Flex Gateway token-claim propagation); without it, actions get attributed to a machine identity. **Net: the agent typically acts as its own scoped service identity; correct scoping is the customer's job.** ([enterprise agent platform](https://engineering.salesforce.com/building-an-enterprise-agent-platform-enforcing-identity-data-and-api-governance/), [MuleSoft traceability](https://blogs.mulesoft.com/agentic-perspectives/why-every-agent-action-must-be-traceable-to-a-human-identity/))
- **Object/field/record security — yes for direct CRM access, with two big holes.** Standard security (CRUD/FLS/sharing rules) applies to the *running user* for Apex/Flow/record grounding; FLS must be explicitly granted per field. **But:** (1) **Apex can run in system context** (`without sharing` / no `stripInaccessible`) and **bypass FLS/sharing**; (2) **UI-layer controls (page layouts, read-only fields) don't protect agents**. ([data security](https://trailhead.salesforce.com/en/data_security/data_security_overview))
- **The Data Cloud/RAG retrieval ACL gap (key finding).** Data Cloud is a **separate data layer**; records ingested into DMOs/search indexes **do NOT automatically inherit the source object's row-level sharing.** A retriever queries the index, not the object under `with sharing`, so **native CRM record-level sharing is not the enforcement mechanism at grounding time** unless explicitly configured. Official docs are conspicuously silent on record-level enforcement during retrieval; Data Cloud uses its own attribute-based policies instead. **Practical mitigation: apply retriever/index-level pre-filters, or ground on native CRM records.** *(Uncertainty: no authoritative Salesforce statement says this verbatim — verify in-org per release.)* ([grounding options](https://trailhead.salesforce.com/content/learn/modules/grounding-an-agent-with-data/review-options-to-ground-an-agent-with-data), [Data Cloud governance](https://trailhead.salesforce.com/content/learn/modules/data-cloud-data-governance-strategies/set-up-your-policy-frameworks))
- **Einstein Trust Layer** wraps every LLM call. Outbound: prompt template → dynamic grounding → **secure data retrieval (pulls only data the user is authorized to access)** → semantic search → data masking → prompt defense → secure gateway. Inbound: **zero data retention** (external LLM "forgets") → toxicity scoring → de-masking → audit trail. ([prompt journey](https://trailhead.salesforce.com/content/learn/modules/the-einstein-trust-layer/follow-the-prompt-journey))
- **CRITICAL nuance: LLM data masking is DISABLED for agents by default.** It applies to embedded generative features, **not** Agentforce agents — so unmasked PII to an agent is *expected behavior*, and you must rely on access controls/FLS/topic scoping instead. ([LLM data masking](https://trailhead.salesforce.com/content/learn/modules/llm-data-masking-in-the-einstein-trust-layer/explore-llm-data-masking))
- **Guardrails & topic scoping.** Reasoning engine can only reach actions **within assigned topics**; platform + Agentforce guardrails (block prompt injection); customer-verification topic for service agents (gates sensitive data until `Verified = true`).
- **The fundamental weakness — the LLM is not a security boundary.** Salesforce's own engineering blog admits the model "interprets language plausibly rather than enforcing rules"; there's no runtime boundary between "employee query text" and "text retrieved from a CRM record." Proven by **ForcedLeak (CVSS 9.4)** — indirect prompt injection via the Web-to-Lead Description field + a CSP hole (an expired allowlisted domain bought for $5) enabling CRM data exfiltration; patched via **Trusted URLs Enforcement** (Sep 2025). Over-provisioning is common (teams grant broad access to "make it work" then leave it). ([ForcedLeak/Noma](https://noma.security/blog/forcedleak-agent-risks-exposed-in-salesforce-agentforce/), [building reliable agents](https://engineering.salesforce.com/building-enterprise-ai-agents-that-are-both-autonomous-and-reliable/))

**RAG design.** Retriever over Data Cloud Search Indexes (Hybrid Search) → augment → generate; structured (CRM objects) + unstructured (Knowledge, PDFs, web) sources; citations via Apex; every retrieval bills Data Cloud (cost driver). Security of retrieval per the **ACL gap above**.

**Multi-role.** Split by audience+environment: customer-facing (Agentforce for Service; Sales/SDR agents) vs employee (internal agents in **Slack**). Role scoping enforced via running-user permissions + topic scoping, not a separate role abstraction.

**Deployment.** Salesforce cloud on **Hyperforce**; LLMs via model gateway (Salesforce + OpenAI + Anthropic-hosted in v3) wrapped by Trust Layer ZDR. Channels: Salesforce app, web, messaging, **Slack**, API. Pricing: **$2/conversation** or **Flex Credits** (~$0.005 each; standard action ~20 credits) + additive Data Cloud cost. *(Data-residency/Hyperforce region guarantees not fully confirmed — verify.)*

**Strengths.** Deepest CRM-native grounding (best if already on Salesforce); enterprise trust scaffolding (Trust Layer ZDR/toxicity/audit); Agent Script determinism targets LLM unreliability; governance/observability maturity in 3.0+ (Command Center, MCP/A2A); reuses battle-tested Salesforce security *when correctly applied*.
**Weaknesses.** (1) **Accuracy gap** — Salesforce's own **CRMArena-Pro** benchmark: ~58% single-turn success dropping to **~35% multi-turn** (~65% failure on realistic multi-step CRM tasks). (2) **Agents lack innate confidentiality awareness** — leak sensitive info unless explicitly instructed, and adding confidentiality instructions *hurts* performance. (3) **Permission model is the core weakness** — LLM enforces nothing; **PII masking off for agents by default**; **Data Cloud retrieval doesn't auto-inherit CRM record sharing**; Apex can bypass FLS; over-provisioning common; prompt injection is architectural (ForcedLeak). (4) Cost unpredictability (consumption + additive Data Cloud). (5) Heavy Data Cloud dependency + data/permission-hygiene prerequisites. ([CRMArena-Pro](https://arxiv.org/abs/2505.18878), [The Register](https://www.theregister.com/2025/06/16/salesforce_llm_agents_benchmark/))

## Cross-cutting analysis — the permission-aware retrieval patterns that matter

**1. The canonical safe pattern (converged across Glean, Amazon Q, Google, Microsoft, ServiceNow):**
- **Ingest ACLs *with* content** (or resolve live), representing each document as an allow-list of **user + (nested) group principals**.
- **Normalize identity into one schema** so a user's references across Slack/SharePoint/Salesforce/etc. collapse to a single resolvable identity + group set. Glean's "own identity schema of groups and memberships" and Amazon's "User Store" are the two clearest implementations; Google leans on `google.subject`/`google.groups` from the IdP; Microsoft leans on Entra being the universal directory already.
- **Authenticate every query (no anonymous), resolve the caller to groups, intersect against stored ACLs — then filter BEFORE the LLM sees anything.** This "filter-before-grounding" ordering is the single most important mechanic: it guarantees the model cannot cite, summarize, quote, or leak content the user can't access, because that content never enters the context window.
- **Keep the source system as a defense-in-depth backstop** (Glean's click-through re-check; Salesforce's native-record `with sharing`).

**2. Identity propagation is where designs differ most — three distinct models:**
- **Run-as-the-end-user (strongest, simplest to reason about):** Microsoft M365 Copilot ("prompt runs in the security context of the user") and ServiceNow `dataAccess` both evaluate every operation against the *invoking* user's live permissions. Nothing to re-map; the platform's existing ACL engine does the work. This is the gold standard when all data lives in one identity domain.
- **Token-exchange federation (strongest for multi-source):** Amazon Q's **trusted identity propagation** (IdP JWT → Identity Center token with `sts:identity_context` → identity-aware SigV4) and Google's **Workforce Identity Federation** (`google.subject`/`google.groups`) both carry a *verified* end-user identity into a retrieval layer that spans many sources. This is the pattern to copy for a cross-system enterprise agent — but its Achilles' heel is **group synchronization**: Amazon only passes groups present *in Identity Center*, not the raw external-IdP groups, so incomplete SCIM sync silently under- or over-matches ACLs.
- **Run-as-a-service-identity (weakest, most footgun-prone):** Salesforce Agentforce agents act as their own "Agent User," and end-user identity propagation is an explicit design burden. This is why Salesforce is the outlier risk: the agent's permissions are decoupled from the human's, so an over-provisioned agent user leaks across roles regardless of who's chatting.

**3. How the best players prevent cross-role leakage — concrete mechanisms:**
- **Per-document principal intersection at query time** (all five ACL-aware players) means a sales rep and a manager asking the identical question get different retrieved sets — role separation is *emergent from ACLs*, not a hand-written rulebook.
- **Filter-before-grounding** stops the LLM from ever holding cross-role data, defeating "summarize what you know about X" leaks.
- **Least-privilege agent/service identities** + **invocation-vs-execution split** (ServiceNow's `securityAcl` vs `dataAccess`) separate *who can invoke* an agent from *whose data it may touch*.
- **Sensitivity labels that flow to outputs** (Microsoft Purview) so derived/generated content inherits the most-restrictive source label.
- **Governance/DSPM layer** to close the **can-access ≠ need-to-know** gap that pure ACL enforcement cannot: oversharing scans, auto-hide, access reviews (Glean Protect+, Purview DSPM, Google sensitive-data scanning, ServiceNow Control Tower).

**4. The universal failure modes to design against:**
- **Source oversharing** — every platform faithfully propagates a too-broad source ACL. ACL-aware retrieval is necessary but *not sufficient*; pair it with oversharing detection.
- **ACL staleness** — most enforce a mirrored snapshot on a crawl/sync cadence (minutes to 24h). A revoked permission can linger. Mitigate with webhooks/incremental identity syncs + a live source re-check on click-through or on high-sensitivity actions.
- **Aggregation/synthesis leakage** — RAG can combine individually-permitted fragments into a sensitive whole. No vendor fully solves this; it's an open risk.
- **Prompt injection as an access-control bypass** — EchoLeak (Microsoft) and ForcedLeak (Salesforce) both show the LLM is *not* a security boundary; untrusted retrieved content can carry instructions. Enforcement must live in the retrieval/permission layer and in output-egress controls (Trusted URLs, Model Armor), never in the prompt.
- **Config burden with silent failure** — Amazon's User Store and Salesforce's agent-user scoping both fail *open and quietly* when misconfigured. Prefer designs that fail closed and surface misconfiguration.

## Implications for the merchantAgent enterprise platform

1. **Adopt filter-before-grounding as a hard invariant.** Retrieve → resolve caller identity + groups → intersect against per-document ACLs → *only then* build the LLM context. Never post-filter a generated answer.
2. **Store ACLs with content** as allow-lists of users + nested groups (the Glean/Amazon/Google model), and **normalize every source's identities into one internal schema** anchored to your IdP via SCIM.
3. **Propagate the real end-user identity** into retrieval (token-exchange like Amazon/Google, or run-as-user like Microsoft/ServiceNow). Avoid the Salesforce trap of a broadly-permissioned shared agent/service identity — if you must use one, scope it to least privilege and inject the end-user's allowed-record filter.
4. **Treat group sync as a first-class correctness concern** — the most cited silent-failure mode across Amazon/Glean/Google is incomplete or stale group membership.
5. **Add a governance/DSPM layer from day one** (oversharing detection, sensitivity labels, access reviews) — ACL enforcement alone will faithfully leak whatever the source overshares.
6. **Assume prompt injection; put access control in the data layer and egress controls, not the prompt.** Constrain tools/actions to the caller's permissions; allow-list output destinations.
7. **Design for role isolation to be emergent from ACLs** (sales vs production vs manager), backed by per-agent scoping and topic/tool gating — not a parallel hand-maintained rule set that can drift.
