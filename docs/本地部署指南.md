# 本地部署指南 —— WSL 后端 + WSL mock + Windows 桌面

> 更新：2026-07-12 ｜ 对应 Windows desktop-local enterprise tool 竖切 ｜ 全本地/reference，零外部依赖（除 LLM 网关）

一句话拓扑：**WSL 里分开跑两个单元**（OpenFGA 授权库 docker 容器 + agentd Go 进程），**Windows 侧编译打包桌面 exe**，桌面连 WSL 里的 agentd（`localhost:8765`，走 WSL2 转发）。

```
Windows 桌面 exe ──HTTP/SSE──> localhost:8765 ──(WSL2 转发/::1)──> WSL agentd(0.0.0.0:8765)
                                                                      │
                                                          ┌───────────┴───────────┐
                                                   OpenFGA(docker :18080)   agentd config/skill + mock ERP/CRM
Windows `%APPDATA%/merchant-agent-desktop/reference-enterprise.db` ── 本地签名 capability executor
```

> `reference-enterprise.db` 是桌面侧的参考制造数据源；它只证明“云端编排/鉴权 → SSE 反向桥 → 本地签名包校验 → 本地执行 → 回执/审计”的拓扑，不是客户 ERP/数据库连接器。真实客户接入仍需专用适配器、凭据管理和字段映射。

## 0. 前置（本机已满足，换机时核对）

- **WSL**：Ubuntu 24.04，装 docker（daemon 在跑）、Go（本机在 `/home/alvin/.local/bin`，`GOROOT=/home/alvin/.local/opt/go1.26.x`）
- **Windows**：Node ≥ 22 + npm
- **网络**：中国网 —— Go 用 `goproxy.cn`、npm/Electron 用 `npmmirror`（下方命令已内置）
- **LLM 网关**：OpenAI 兼容地址 + api_key（本机用 `https://www.linx2.ai` / `gpt-5.5`）

## 2. WSL —— 起 OpenFGA（mock 授权库）

```bash
cd /mnt/d/merchantAgent/backend
docker compose up -d                     # OpenFGA -> host :18080（8080 被本机占用，故用 18080）
# 等就绪：
curl -fsS http://localhost:18080/healthz # {"status":"SERVING"}
```
> 内存 datastore，重启即清空；agentd 每次启动都会重新播种，无需手动初始化。healthcheck 可能显示 unhealthy（gRPC 探针），不影响 HTTP API。

## 3. WSL —— 编译 + 运行 agentd

```bash
export PATH="/home/alvin/.local/bin:$PATH"
export GOPROXY=https://goproxy.cn,direct GOSUMDB=off      # 中国网必需
cd /mnt/d/merchantAgent/backend
go build -o ~/merchantagent/agentd ./cmd/agentd           # 编到 home，别在 /mnt 跑
# 载入配置并后台常驻（detached，退终端不掉）：
set -a; . /mnt/d/merchantAgent/backend/dev.env; set +a
setsid -f ~/merchantagent/agentd </dev/null >~/merchantagent/agentd.log 2>&1
sleep 2; cat ~/merchantagent/agentd.log                   # 见 "listening on 0.0.0.0:8765"
```

可选自测（连真 OpenFGA 跑全测）：`go test ./...`（11 包应全绿）。

## 4. Windows —— 编译 + 打包桌面 exe

在 Windows 侧（git-bash / PowerShell 均可，下面是 bash）：

```bash
cd /d/merchantAgent/desktop
export ELECTRON_MIRROR=https://npmmirror.com/mirrors/electron/
export ELECTRON_BUILDER_BINARIES_MIRROR=https://npmmirror.com/mirrors/electron-builder-binaries/
export CSC_IDENTITY_AUTO_DISCOVERY=false                  # 本地不签名

npm install --registry=https://registry.npmmirror.com     # 中国网必需；勿中途 kill（见坑2）
npm run build                                             # electron-vite 三端打包 -> out/
npm run dist:dir                                          # 验收用免安装目录（含原生模块重建）
npm run dist:win                                          # electron-builder 出 NSIS 安装包
```

产物在 `desktop/dist/`：
- `merchantAgent Setup 0.1.0.exe` —— NSIS 安装包（≈82MB）
- `win-unpacked/merchantAgent.exe` —— 免安装版

`dist:dir` 必须包含并保持与源码同内容：

- `dist/win-unpacked/resources/capabilities/reference-manufacturing.cap.json`
- `dist/win-unpacked/resources/capabilities/reference-public.pem`
- `dist/win-unpacked/resources/app.asar.unpacked/node_modules/better-sqlite3/build/Release/better_sqlite3.node`

源码 capability 位于 `desktop/resources/capabilities/`。私钥不随应用分发；仓库只含已签名包和验签公钥。

`@electron/rebuild` 是直接开发依赖。打包脚本会先用 `electron-rebuild -f` 把工作区 native module 强制切到 Electron ABI；`npmRebuild:false` 只关闭 electron-builder 冗余且非强制的重建路径。`win-unpacked` 可能通过硬链接复用该 binary，因此矩阵顺序必须是 test/typecheck/build → `dist:dir`。之后若执行 `npm rebuild better-sqlite3` 恢复 Node ABI，交付前必须再跑一次 `dist:dir`。

### 打包已知坑（本机已踩平，换机可能复现）

1. **winCodeSign 符号链接权限失败**（`Cannot create symbolic link ... darwin/...dylib`）：Win10 无 Developer Mode/管理员时无法建符号链接，而那是 macOS 文件、Windows 打包用不上。预解压排除 darwin 即可：
   ```bash
   C="$LOCALAPPDATA/electron-builder/Cache/winCodeSign"   # 或 ~/AppData/Local/...
   7za x "$C"/*.7z -o"$C/winCodeSign-2.6.0" -xr!darwin -y  # 用 node_modules/7zip-bin 里的 7za
   # 之后再跑 dist:win，electron-builder 会直接用这份缓存、跳过它自己会失败的解压
   ```
2. **npm install 被中断**会留下残缺 node_modules（`7zip-bin`/`app-builder-bin` 空目录、`.bin` 缺 shim）→ 重新完整 `npm install`（走 mirror）补齐。

## 5. 启动、参考库 reset 与验收

装好/解压后双击 `merchantAgent.exe`（**先保证 §2、§3 的 OpenFGA + agentd 在跑**）。切换顶部身份问 “SO-1001 进度/利润/齐套”，看**同问不同权**。

需要把生产进度恢复为种子值 60% 时，先退出桌面 App，仅删除下面一个文件：

```powershell
Remove-Item -LiteralPath "$env:APPDATA\merchant-agent-desktop\reference-enterprise.db" -Force
```

不要删除 `$DATA_DIR/config.db`、`$DATA_DIR/skills.db`、整个 `%APPDATA%\merchant-agent-desktop` 目录或 capability 文件。桌面下次启动会重建参考库；配置和 Skill 分配保持不变。

参考 capability 只支持两个 allowlisted 工具：

| 工具 | 风险/行为 |
|---|---|
| `query_order_status` | 本地只读；只返回订单、工单、状态、交期、完成率、备注、版本，不返回成本/SQL/路径 |
| `report_production_progress` | `low_write`；必须显示 变更前→变更后 预览并由用户确认；带幂等键、乐观版本冲突和写后校验 |

Windows 验收顺序：

1. `u_sales1` 查询 `SO-1001`，应为 60%/version 1 且无成本字段；请求更新应被能力门禁拒绝，不出现本地确认。
2. `u_plan` 查询后请求更新至 80%，确认原生对话框显示 `SO-1001`、`WO-1001`、`60% → 80%`，再点“确认写入”；结果应为 80%/version 2。
3. 切回 `u_sales1`，应读到 80%/version 2。以 `u_boss` 请求 `/audit`，确认哈希链 `verified=true`，且 read/write/read-back 条目完整，write 带 `confirmed=true` 和 before/after。
4. 以 `u_boss` 打开 配置→分配，取消 `production-progress` 的“计划员”；新 planner 会话应立即失去更新能力。此步骤会持久修改 Skill 配置，不要用删除配置库的方式“恢复”。
5. 在桌面尺寸和 390px 窄窗各检查一次：无水平滚动、遮挡或按钮/文本越界。

真实模型可能在没有可用写 Skill 时直接用文本拒绝，而不产生工具调用；这种情况 UI 拒绝成立，但审计不会凭空生成 tool-deny。强制工具尝试后的 deny/no-local-request/audit-denied 路径由 `backend/cmd/agentd/localtool_vertical_test.go` 确定性覆盖。

### Gate A 与 Gate B 独立验收探针

正常拒绝可能停在 Gate A（Skill 未向 sales 暴露）。要真机验证 Gate B（记录级关系），以 `u_boss` 通过 `/admin/skills/production-progress` 暂时把 roles 从 `["manager_tier"]` 改为 `["manager_tier","sales"]`，只绕过 Skill 暴露门禁；随后以 `u_sales1` 请求更新 `SO-1001`。预期服务端以 `business_record#operator` 拒绝，桌面不收到 `local_tool_request`、不显示原生确认，参考库时间戳和内容不变。

以 `u_boss` 请求 `/audit`，确认 `verified=true`，并有 `tool=report_production_progress`、`decision=deny`、`status=denied`、`confirmed=false`、`reason="no operator access to business_record order/SO-1001"` 的真实条目。探针结束后必须立即把 roles 恢复为 `["manager_tier"]`，再读回 `/admin/skills/production-progress` 确认恢复；不要重置或删除 Skill/config 数据库。2026-07-12 的实机探针已按此流程完成并恢复。

命令行冒烟（可选）：
```bash
curl -X POST http://localhost:8765/login -d '{"userId":"u_sales1"}'
curl -N -X POST http://localhost:8765/chat -H 'content-type: application/json' \
  -d '{"sessionId":"s","userId":"u_smgr","question":"SO-1001 的利润是多少"}'
```

**网络说明（重要）**：桌面连的是 `localhost:8765` 而非 `127.0.0.1:8765`。本机 WSL2 的 localhost 转发是**纯 IPv6**（`.wslconfig` 的 `mirrored` 在 Win10 被忽略），`127.0.0.1` 字面量连不上、`localhost`/`::1` 可达；代码已用 `localhost`（Node fetch autoSelectFamily 自动选通的族），勿改回。

**身份说明（重要）**：`u_sales1` / `u_smgr` / `u_boss` / `u_fin` / `u_plan` 都来自 mock 组织，桌面身份下拉会直接提交 `userId`。它不是登录或安全边界；生产必须由企微 OAuth/JWT 会话派生身份。

## 6. 重启（WSL 重启或关机后）

```bash
# WSL 里，按顺序：
cd /mnt/d/merchantAgent/backend && docker compose up -d
set -a; . /mnt/d/merchantAgent/backend/dev.env; set +a
setsid -f ~/merchantagent/agentd </dev/null >~/merchantagent/agentd.log 2>&1
# 然后 Windows 双击 merchantAgent.exe
```
桌面 exe 已装好就不必重打；只有改了 `desktop/` 源码才需重跑 §4。
