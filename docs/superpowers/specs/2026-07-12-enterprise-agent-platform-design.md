# 跨行业企业 Agent 平台设计

> 版本 v1.0 | 2026-07-12 | 状态：设计已确认，待实施计划

## 0. 文档目的

本设计重新界定产品方向，并补齐现有 Phase 0/Phase 1 文档未覆盖的企业数据接入与低风险写操作。

现有实现仍是贸易/制造 mock 竖切；本设计把产品定位提升为**跨行业企业 Agent 平台**。贸易/制造是第一个行业能力包，不是平台核心的数据模型。

本设计不否定现有权限、Skill、连接器和桌面客户端方向，而是作出以下修正：

- Windows App 本地执行器作为首版企业侧工具执行主路径。
- 企业集中网关与云 API 连接器保留为可选执行位置，不是每家企业的前置要求。
- Skill、可执行工具、凭据和权限在内部必须分离，即使管理界面把它们呈现为一个能力包。
- 首版支持读取与低风险写入，不开放任意 SQL 或高风险业务写入。
- 平台核心不硬编码订单、成本、案件等行业概念。

## 1. 已确认的产品决策

1. 产品是跨行业企业 Agent 平台，贸易/制造是首个验证行业。
2. 采用“通用底座 + 首条行业竖切”，不先建设完整无代码平台。
3. 首期由我们的实施工程师配置连接器和业务工具；企业管理员管理角色、Skill 和数据权限。
4. 采用混合部署：云端负责 Agent 编排、身份、授权、配置和审计；企业侧执行实际工具。
5. 首版企业侧执行以员工 Windows App 内的本地执行器为主。
6. 云端模型可接收用户问题及本次任务必要、已授权、已裁剪的业务字段。
7. 数据库凭据、完整数据、无关字段和任意 SQL 不上云。
8. 首版支持读取与低风险写入；高风险写入禁止。
9. 首版只有企业管理员可以安装、发布和分配 Skill，普通主管不能自行扩权。

## 2. 设计原则

### 2.1 平台核心行业中立

平台核心只认识以下通用概念：

- tenant：企业租户；
- principal：当前用户；
- role：组织角色；
- resource：业务资源；
- action/tool：可执行动作；
- skill：完成业务任务的剧本；
- data domain：敏感数据域；
- policy decision：授权判断；
- execution/audit：执行与审计。

`order`、`work_order`、`case`、`document` 等属于行业能力包，不应继续硬编码进平台核心。

### 2.2 Skill 不等于执行能力

内部语义固定为：

```text
Skill = 如何完成任务 + 允许组合哪些工具
Tool/Connector = 实际能连接什么、执行什么
Authorization = 当前用户能否调用、能碰哪些记录和数据
```

产品界面可以把三者包装为一个“企业能力包”，但不能让 Skill 同时承载提示词、任意代码、数据库凭据和安全边界。

### 2.3 不给模型任意数据库能力

禁止向模型暴露以下通用能力：

```text
execute_sql(sql)
query_table(table, where)
update_table(table, fields)
```

只允许经过审核的业务语义工具，例如：

```text
query_order_status(orderId)
report_production_progress(workOrderId, completionRate, note)
query_case_status(caseId)
archive_case_document(caseId, documentId, category)
```

## 3. 平台分层

| 层 | 职责 | 复用范围 |
|---|---|---|
| 员工入口 | Windows App、登录、聊天、确认操作、本地执行 | 跨行业复用 |
| 云端平台核心 | Agent 编排、身份、权限、Skill、租户配置、审计 | 跨行业复用 |
| 行业能力包 | 业务资源、工具合同、Skill、权限模板、风险等级 | 按行业提供 |
| 企业侧执行器 | 调用 API/数据库/网页、保管本地凭据、裁剪结果 | 运行时复用，配置不同 |
| 企业数据绑定 | 字段映射、参数化操作、账号绑定、业务规则 | 每家企业配置 |

新增企业时主要替换企业数据绑定；新增行业时主要新增行业能力包。平台核心、客户端、权限和审计保持稳定。

## 4. 工具执行拓扑

平台使用统一工具合同，支持三种执行位置。

### 4.1 Windows App 本地执行器（首版主路径）

云端 Agent 产生结构化工具调用后，通过现有桌面反向桥下发到员工 Windows App。本地执行器验证调用并使用员工电脑可访问的企业资源执行。

适用场景：

- 员工电脑能访问企业 ERP/API/数据库；
- 使用员工现有 ERP 会话、Windows/AD 身份或个人数据库账号；
- 实时查询、低风险写入、本地文件和网页操作；
- 不要求后台无人值守执行。

### 4.2 企业集中网关（可选）

企业数据库不允许员工电脑直连、需要共享受限凭据、后台任务或批量操作时，在企业内网安装一次集中执行服务。它主动出站连接云端，不要求数据库开放公网入口。

### 4.3 云 API 连接器（可选）

企业系统提供安全、可公网访问的官方 API 时，云端连接器直接调用，无需桌面执行器或集中网关。

Skill 不依赖执行位置。同一工具合同可由桌面、网关或云端连接器实现。

## 5. 企业能力包

企业管理员看到的一个能力包，内部包含：

```text
企业能力包
├─ Skill：Agent 如何完成业务任务
├─ 工具清单：允许调用哪些读写动作
└─ 本地连接器：这些动作如何访问企业系统
```

每个工具合同至少声明：

- 工具 ID、版本和描述；
- 输入参数与校验规则；
- 允许返回的字段；
- 业务资源类型和资源 ID 参数；
- 涉及的数据域；
- 风险等级；
- 是否需要本地确认；
- 超时、结果数量、幂等与重试规则；
- 执行位置要求；
- 回读验证方式。

能力包必须租户隔离、版本化和签名，支持测试、发布、停用与回滚。数据库密码、API 密钥和个人令牌不包含在能力包内。

首版能力包及本地连接器由我们的实施工程师创建和测试；企业管理员不直接编写 SQL。成熟后的低风险、重复配置再逐步产品化。

## 6. 身份与权限

### 6.1 三层身份和权限

1. 企微或其他 IdP 提供企业、用户、部门、主管、职位和标签。
2. 平台把组织信号映射为角色，并决定可用 Skill、工具和数据域。
3. 企业系统账号继续限制员工在 ERP、案件系统或数据库中的实际权限。

最终权限是交集：

```text
平台角色与 Skill 权限
∩ 平台记录级和数据域权限
∩ 企业系统账号实际权限
```

任意一层拒绝都不得执行。

### 6.2 企业系统账号绑定

优先级如下：

1. 员工自己的 ERP/API 登录令牌；
2. Windows/AD 集成身份；
3. 员工自己的数据库账号；
4. 企业配置的最小权限服务账号。

企微只能提供身份与组织关系，不能自动导入 ERP 网页内部的权限。需要通过账号绑定、企业账号映射、源系统 ACL 同步或本地工具显式过滤补齐。

直接访问数据库时，不能假设 ERP 应用层权限仍会生效。工具必须显式带当前用户和资源范围，并使用参数化操作、固定字段白名单和最小权限账号。

### 6.3 管理员分配

Skill 与角色是多对多关系：

- 一个角色可以拥有多个 Skill；
- 一个 Skill 可以分配给多个角色；
- 行业模板只提供建议角色，不自动授权；
- 首版只有企业管理员可以安装、分配、撤销和发布 Skill。

Skill 分配是能力权限；数据域和记录级授权是数据权限，两者继续独立。

## 7. 数据外传边界

混合部署下，云端可接收：

- tenant ID、稳定 user ID、角色和授权关系；
- 用户提交的问题；
- 本次任务必要且已经过权限过滤的工具结果；
- 工具调用与执行审计元数据。

企业侧执行器必须执行：

- 字段裁剪：无关字段不返回；
- 权限过滤：无权字段和记录不返回；
- 结果限量：禁止整库或无限结果集上云；
- 可选脱敏：手机号、证件号等按任务需要遮盖。

以下内容不得上云：

- 数据库和 API 凭据；
- 完整数据库、备份或无关表数据；
- 模型生成的任意 SQL；
- 当前任务不需要的敏感字段。

用户输入本身也可能包含敏感内容，因此后续生产化仍需数据分级、模型端点、日志留存和企业策略配置。

## 8. 低风险写操作

### 8.1 首版风险分级

| 等级 | 示例 | 首版行为 |
|---|---|---|
| 只读 | 查询状态、查询案件 | 允许，不需要写确认 |
| 低风险写入 | 更新进度、添加备注、标签、资料入档 | 允许，每次确认、回读、审计 |
| 高风险写入 | 改价、财务入账、删除、法律文件定稿 | 禁止 |

### 8.2 状态流转

```text
proposed
→ awaiting_confirmation
→ executing
→ verifying
→ succeeded | failed | unknown
```

写入前，Windows App 必须展示业务资源、修改前后值、执行身份和数据源，由员工明确确认。

每次写入必须带幂等键。相同幂等键重复提交不得重复修改。执行完成后必须回读验证。

`unknown` 表示调用超时且无法确认是否已写入。此状态不能自动重试，必须先查询实际数据再决定下一步。

### 8.3 错误分类

客户端至少区分：

- 没有权限；
- 本地连接器未安装；
- 企业系统未登录或凭据失效；
- 参数校验失败；
- 源数据已被他人修改；
- 企业系统拒绝；
- 执行明确失败；
- 执行结果未知。

## 9. 审计

每次工具调用记录：

- 企业、用户、角色和设备；
- Skill、工具和版本；
- 业务资源 ID；
- 允许审计的修改前后字段；
- 用户确认时间；
- 本地执行编号与幂等键；
- 权限判断；
- 执行、回读和最终状态。

普通员工可以查看自己的操作记录；企业管理员可以查看本企业完整审计。审计可见不等于敏感业务字段自动可见，展示仍需数据权限过滤。

## 10. 首条交付竖切

### 10.1 目标

用贸易/制造参考企业验证：同一个云端 Agent 能按角色把受控工具下发到员工 Windows App，本地读取和低风险写入企业数据，并让其他岗位读取到更新后的权威状态。

### 10.2 参考能力包

```text
订单状态查询
  工具：query_order_status
  角色：销售、生产、主管

生产进度上报
  工具：report_production_progress
  角色：生产、主管
```

本地参考企业数据库先使用可写 SQLite 模拟企业系统，但工具合同和反向桥按未来真实 API/数据库连接设计。

### 10.3 验收闭环

1. 管理员给销售分配订单状态查询。
2. 管理员给生产分配订单状态查询和生产进度上报。
3. 销售查询 `SO-1001`，看到生产进度 60%。
4. 销售尝试更新进度，被能力权限拒绝。
5. 生产员工提出把关联工单更新为 80%。
6. Windows App 展示变更预览，员工确认。
7. 本地连接器写入、回读并返回成功。
8. 销售重新查询同一订单，看到进度 80%。
9. 普通销售看不到成本等未授权字段。
10. 企业管理员看到查询、拒绝、确认、写入和回读的完整审计。

### 10.4 异常验收

- 本地连接器未安装时拒绝执行；
- 数据源未配置或凭据失效时明确提示；
- 用户取消确认时不写入；
- 同一幂等键重复提交时只写一次；
- 源数据发生并发修改时返回冲突；
- 写入超时时进入 `unknown`；
- Skill 撤销后立即失效；
- 工具包签名或版本不匹配时拒绝运行。

## 11. 首条竖切明确不做

- 真实企微 OAuth 和通讯录同步；
- 企业自助编写 SQL 或连接器；
- 高风险写操作和主管审批；
- 企业集中网关；
- 法律等第二行业能力包；
- 后台无人值守任务；
- 真正客户生产数据库接入。

## 12. 后续拆分

首条竖切完成后，分成独立项目继续设计和实施：

1. 真实企业连接器接入与实施工具链；
2. 真实企微身份、组织与企业系统账号绑定；
3. 企业集中网关与后台执行；
4. 高风险写操作、审批与补偿；
5. 法律行业能力包，用于验证核心不需要修改；
6. 将成熟的工程师配置逐步开放为受限自助能力。

## 13. 与现有文档的关系

- `research/10-方案综合-企微多租户落地设计.md` 的跨行业连接器、企微组织和权限方向继续有效。
- `research/04-mcp-and-skills.md` 中“MCP 连接、Skill 指导”的分层继续有效。
- `docs/phase1-设计.md` 和 M1-M6 仍是已经完成的 mock 技术验证，不代表真实企业接入完成。
- 本设计取代“生产连接器默认必须使用企业集中网关”的单一路径假设，改为桌面本地执行优先、网关与云 API 可选。
- 后续实现计划必须以本设计的首条交付竖切为边界，不把长期平台能力一次性塞入同一里程碑。
