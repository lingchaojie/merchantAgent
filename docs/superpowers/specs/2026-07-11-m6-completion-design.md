# M6 管理界面收口设计

> 版本 v1.0 | 2026-07-11 | 补充 `2026-07-10-m6-management-ui-design.md`

## 0. 目标

补齐现有 M6 管理界面与原设计之间的差距，使租户管理员能在桌面端完整完成角色、职位映射、技能、能力分配和数据域授权配置，并能在浏览器 mock 中重复验收同一套交互。

本次沿用现有 `config`/`skill` SQLite 真源、`wire.Projector`、`/admin/*` REST API、通用 admin IPC 和 5 个管理 Pane。后端数据模型与路由不变。

## 1. 范围

### 1.1 本次完成

- 角色：新建、修改名称/描述、删除。
- 职位映射：增删、修改、上移、下移、整体保存。
- 技能：空白新建、从模板克隆、修改全部可编辑字段、删除。
- 分配：继续保持技能到角色（能力闸 A）与数据域到角色（数据闸 B）两套独立操作。
- 管理入口：前端根据当前身份探测管理员权限；非管理员入口禁用并提示，后端 `requireAdmin` 仍是最终安全边界。
- 浏览器 mock：提供合法、完整、会话内持久化的角色/规则/技能/模板/数据域状态和 CRUD 行为。
- 测试与文档：补齐自动化测试、浏览器验收，并统一进度文档中的 M6 状态。

### 1.2 不做

- 不解决配置 SQLite 与 OpenFGA 投影之间的跨存储事务一致性。
- 不接真实企微 OAuth，不改变 demo 身份注入方式。
- 不增加 tag 派生角色管理。
- 不实现研究文档中的制造场景 M6“排产建议/工单重排”。
- 不引入新的前端运行时或测试依赖。
- 不重构为通用 CRUD 框架。

## 2. UI 行为

### 2.1 角色

`RolesPane` 保留紧凑列表。每行提供编辑和删除操作；编辑态允许修改 `label` 与 `description`，`roleId` 不可变。新建表单包含 `roleId`、显示名和可选描述。

成功写入后重新调用 `listRoles()`；失败时保留当前输入并显示服务端错误。删除继续由后端级联清理职位规则、技能角色和数据域角色授权。

### 2.2 职位映射

每条规则继续使用逗号分隔关键词和角色下拉。新增上移/下移按钮；第一条禁用上移，最后一条禁用下移。规则数组顺序就是 `ReplaceRules` 的保存顺序和运行时第一命中优先级。

保存前过滤空关键词；保存成功后重新读取规则并显示“已生效”，失败时保留本地编辑内容。

### 2.3 技能

技能页提供两条创建路径：

1. 从平台模板克隆。
2. 新建空白技能，填写不可变的 `skillId` 和必填名称。

编辑器覆盖 `name`、`description`、`playbookMd`、`allowedTools`、`dataDomains` 和 `roles`。工具来自 `/admin/tools`，数据域来自 `/admin/domains`，角色来自 `/admin/roles`，不再使用硬编码数据域列表。

技能可删除。保存、克隆、新建或删除成功后重新读取技能列表；失败时显示服务端错误。

### 2.4 分配和数据域

`AssignPane` 保持两块清晰分离的矩阵：

- 闸 A：修改 skill 的 `roles`。
- 闸 B：增加或删除 `role:<tenant>/<roleId>#assignee` 数据域 grant。

`DomainsPane` 保留原始 subject 授予能力。所有写操作继续串行化，写入成功后重新读取服务端状态。

### 2.5 管理入口

`App` 在当前 `userId` 变化时通过 `listRoles()` 探测管理员权限。探测中和无权限时配置按钮禁用；无权限时 tooltip 说明需要管理员身份。管理员点击后进入 `AdminView`。

这个前端状态只改善体验，不作为安全判断。所有 `/admin/*` 请求仍由后端 `requireAdmin` 检查 OpenFGA `tenant.admin`。

## 3. 浏览器 Mock

把当前固定响应改为模块内存状态，并提供创建 mock admin handler 的工厂函数，便于每个测试获得独立状态。

初始状态与后端 seed 的结构对齐：

- 至少包含现有角色、职位规则、`order360`/`customer360` 技能和 `order-360` 模板。
- 数据域为 `cost`、`pricing`。
- 角色 subject 使用完整格式 `role:mock-corp-001/<roleId>#assignee`；用户 subject 保持 `user:<id>`。

mock 实现所有当前 admin client 调用：角色 CRUD、规则整体替换、技能空白创建/模板克隆/修改/删除、grant 增删。非管理员始终返回 403。mock 只在浏览器预览存活，不写文件、不连接 OpenFGA。

## 4. 错误与状态

- 每个 Pane 只显示本 Pane 最近一次失败，不吞掉主要加载或写入错误。
- 写按钮在请求进行中禁用，防止重复提交和丢失更新。
- 成功响应代表后端 `Reproject` 已完成；UI 随后重新读取真源并显示短暂“已生效”。
- 删除不存在对象、非法 id/subject、非管理员和投影失败沿用后端错误文本。

## 5. 测试

遵循现有 Vitest 和 Go 测试体系，不增加依赖。

### 5.1 自动化

- 为规则移动纯函数写测试：边界不移动，中间项能上下移动，原数组不被修改。
- 扩充 admin client 测试：角色更新、空白技能创建、技能删除和 grant 删除的 method/path/body 正确。
- 为 mock admin 工厂写测试：非管理员 403；角色修改可回读；规则顺序可持久化；模板克隆与空白技能创建可回读；合法 grant 可增删。
- 保持现有主进程 admin 代理、后端 handler、Projector 和 OpenFGA 撤权测试全绿。

### 5.2 浏览器验收

使用管理员身份完成以下闭环：

1. 新建角色，修改其名称和描述，再删除。
2. 调整职位规则顺序并保存。
3. 新建空白技能，编辑全部字段并删除。
4. 从模板克隆技能。
5. 在分配页分别切换闸 A 和闸 B。
6. 在数据域页增加和删除合法 subject。
7. 切到非管理员，确认配置入口不可用；直接 API 请求仍返回 403。

桌面生产构建还需通过 `npm test`、`npm run typecheck` 和 `npm run build`。后端需显式连接 OpenFGA 运行 M6 相关集成测试。

## 6. 完成标准

以下条件同时满足后，M6 才能标记完成：

- 原 M6 spec 中角色、职位映射、技能、分配、数据域的桌面操作全部可达。
- 浏览器 mock 与真实 API 使用相同数据形状和 subject 规则。
- 自动化测试、类型检查、桌面构建、OpenFGA 集成测试全部通过。
- 浏览器验收剧本通过，无控制台错误或明显布局溢出。
- `docs/实现进度.md` 的顶部、里程碑表和 M6 章节状态一致。
